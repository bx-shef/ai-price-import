## 1. Цель

Переделать AI-импорт прайсов (`procure-ai`) с текущей 4-сервисной раскладки на
**единый Nuxt 4 + Nitro монолит** по образцу `client-bank-alfa-by` и превратить из внутреннего
инструмента под один портал в **облачное приложение для Маркета Bitrix24** (мультитенант, OAuth).
Сохраняем ценность (агент + промпт + MCP-инструменты + ядро извлечения текста) и добавляем то,
чего нет: дисциплину чистого ядра с тестами, очередь под нагрузку, публичный лендинг, маркетинг.
**Убираем весь код для ядра/коробки Б24** — вся интеграция через MCP поверх стандартного REST.

**Что импортируем:** любой **документ с табличной частью** (накладная/счёт/КП/прайс/спецификация —
не только «счёт»). Суть действия: **найти контрагента и внести товары** в целевую сущность; тип
цели — по правилам роутинга. **Языки документов: русский, белорусский, казахский** — подробно с
примерами/выводами в [`06-multilingual.md`](06-multilingual.md).

**Критерий готовности MVP редизайна:** менеджер на портале, где установлено приложение, загружает
документ с табличной частью → агент через изолированный MCP (стандартный REST по OAuth-токену)
создаёт целевую сущность (по правилам роутинга) с товарами; лендинг + листинг Маркета опубликованы;
деплой одним пайплайном (GHCR + Watchtower).

## 2. Целевая архитектура (одной фразой)

Nuxt-монолит: **лендинг + in-portal UI + Nitro-backend** в одном приложении; backend
оркеструет **агента Claude Code**, который ходит в Bitrix24 **только через изолированный MCP**;
MCP зовёт **стандартный REST Bitrix24** (ноль кода в коробке). Тяжёлые шаги (извлечение текста, прогон агента, запись в CRM) —
через **очередь BullMQ + Redis**. Детали — [`02-target-architecture.md`](02-target-architecture.md).

| 3 | **Настройки/маппинг портала** — форма настроек + чистое ядро маппинга: каталог, **поле артикула поставщика** (тип текст=построчно / строка=разделитель, задаёт админ), **целевая сущность + направление** (сделка/смарт-процесс/инвойс/КП; **роутинг**: ручной override → упорядоченные правила `{match: type|keyword → target}` → дефолт; `resolveTarget` — чистая функция), **тумблер сохранения файла** (дело — настраиваемое `crm.activity.configurable.add`, `layout` от приложения), **стратегия поиска/создания товара**, **единицы** (словарь→`catalog.measure` + дефолт + авто-создать), **чат ошибок + чат уведомлений**. НДС — ставки из Б24 (`crm.vat.list`). Налоговый ID — фикс. `RQ_INN`. Договор — не тут (Q8) | 📝 |
| 4 | **Лендинг + маркетинг** — тёмная бренд-оболочка, `landing.ts`, HeroGraph, BriefForm, Метрика, OG, листинг Маркета | 📝 |
| 5 | **Ядро загрузки/извлечения** — `app/utils` парсинг + `server/` извлечение текста (pdftotext/OCR/office) в очереди; UI загрузки **только in-portal** (веб+мобильное приложение) с ручным выбором цели у файла; результат = статус + **ссылка «открыть»** сущность. Standalone-загрузка — кастом (маркетинг) | 📝 |
| 6 | **Агент + изолированный MCP** — порт промпта и инструментов (стандартный REST по токену портала) как первоклассный код с тестами; промпт распознаёт налоговый ID **с учётом страны и языка** (ИНН/УНП/БИН…, документ на любом языке → поиск по `RQ_INN`) | 📝 |
| 7 | **Запись в CRM (стандартный REST)** — резолв цели (ручной→авто→дефолт) → `crm.item.add` + `crm.item.productrow.set` (НДС: ставки из `crm.vat.list`, `taxIncluded` единый на документ, ставки по позициям могут различаться); файл на **общий Диск** (папка приложения → подпапка по месяцам) + **настраиваемое дело** `crm.activity.configurable.add` с файлом (`layout` от приложения; тумблер сохранения файла). Без коммента/UF-поля. Lookups по маппингу (несколько совпадений → мин. ID + предупреждение); **без дедупа** (всегда создаём); **матрица нестыковок** (поставщик/товар/валюта/единица) → предупреждение или **чат ошибок**; предпросмотра нет | 🧪 ядро: `crmWrite`/`companyLookup`/`disk`/`configurableActivity` + тесты; НДС ✅ вживую (не связано: нет worker/агента) |
| 8 | **Метрики, обратная связь, оператор** — метрики 2 уровня (мотивирующие всегда на `/app` + детальные `/metrics`); **отзывы сотрудников** (👍/👎/💡+текст) → **закрытый репо издателя** (outbox) + **согласие на файл** (политика данных); **сигналы агента**; **страница оператора как в эталоне** (авторизация + `/queues` ECharts); health, AI-телеметрия, политика диагностики | 📝 |
| 9 | **Деплой + CI/CD** — Docker multi-target, CSP-hashing, GHCR + Watchtower, required-check `ci` | 📝 |

Лендинг (этап 4) не зависит от ядра — можно вести параллельно. **Этапа деплоя PHP-контроллеров
больше нет** (нет кода в коробке).

**Метрики/обратная связь/оператор** (этап 8): метрики 2 уровня (мотивирующие на `/app` + детальные
`/metrics`); отзывы сотрудников (👍/👎/💡) + сигналы агента (уверенность/проблемы → ручной разбор,
алерты дрейфа); служебная зона оператора как в эталоне (login + `/queues`). Детали —
[`02-target-architecture.md`](02-target-architecture.md) §7. Также аргумент ценности в маркетинге.

## 5. Что переносим из старого проекта

| Актив | Куда в новой раскладке | Примечание |
|---|---|---|
| Промпт `prompts/main.md` | `prompts/` или `server/agent/` | Ценность как есть; ревизия правил (НДС→штатный, единицы) |
| Логика инструментов (поставщик/товар/сделка) | новый MCP-пакет `mcp/tools/*` | **Переписать на стандартный REST** (`crm.*`/`catalog.*`) вместо кастомных `shef:purchase.api.*`. **Поиск договора не переносим** (Q8) |
| Ядро извлечения текста | `server/` + утилиты | pdftotext/tesseract/office; в очереди |
| Homoglyph-логика | `app/utils/*` (чистая, тесты) | Уже почти чистая функция |
| OAuth/мультитенант из старого MCP | `server/utils/*` + MCP | **Оживает**: per-portal токены, `ensureAccessToken`, `makePortalRestCall` (как эталон) |
| Метрики + **обратная связь** | `server/` + Postgres | Пересобрать: метрики 2 уровня; отзывы сотрудников (👍/👎/💡) + сигналы агента |
| Авторизация оператора | `server/utils/session.ts` + `/login` | Модель эталона (HMAC-cookie, rate-limit) для служебной зоны |

**Не переносим:** весь `b24-controller/` (PHP-модуль коробки, `procure*.php`, **патч ядра НДС** —
решение Q7); мёртвый чат-бот (#241); вендоренный `mcp/` как git-subtree; хардкод инфоблоков/полей
(заменяем пер-портальным маппингом); i18n на 19 локалей (только `ru`, перевод позже);
**eval-харнесс** (убран — вместо скоринга требование 1-в-1, см. ниже).

### Детальные решения (обсуждение документов)

| # | Решение |
|---|---|
| D1 | **Монорепо** — MCP не выносим в отдельный репозиторий, живёт в этом же (`mcp/`). |
| D2 | **Worker внутри backend-процесса** в MVP; выделенный worker-контейнер — под нагрузкой. |
| D3 | **Postgres** — OAuth-токены, идемпотентность джоб, метрики; Redis — только очереди. (Бизнес-дедупа импорта нет — см. решение по дубликатам.) |
| D4 | **Локализация только `ru`**; i18n-инфраструктуру и переводы добавим позже. |
| D5 | **Eval-скоринг-харнесс убран.** Требование точности — позиции и цены **1-в-1 со счётом**. Проверка — юнит-тесты чистого ядра + **golden-фикстуры** «текст документа → ожидаемые productrow» (дешёвый регресс-guard без скоринга; ловит дрейф извлечения при смене промпта/модели — рекомендация ревью). |

## 7. Риски и рекомендации (из ревью 5 специалистов)

- **✅ СНЯТ вживую — НДС без патча ядра работает.** Проверено на `B24_HOOK`: `crm.item.productrow.set`
  с `price=100, quantity=2, taxRate=22, taxIncluded=Y` (BYN) → `priceBrutto=100`, `priceNetto=81.97`,
  **сделка: Итого=200.00, НДС=36.07** (200×22/122 ✓). Патч ядра НЕ нужен — он был нужен для **legacy**
  `crm.deal.productrows`, а универсальный `crm.item.productrow.*` считает НДС сам. Тезис Q7 «ноль кода
  в ядре» подтверждён. Бонус: **free-form позиция** (без `productId`, только `productName`) — работает.
- **Рекомендация: вертикальный спайк раньше инфраструктуры.** Сквозной срез
  upload→extract→агент→MCP→`crm.item.add`+`productrow.set` на живом портале (один хардкод-портал,
  одна тест-накладная, без очереди/мультитенанта/лендинга) — проверить связку и фидельность до
  постройки этапов 2–5 вокруг неё. Самое непроверенное — AI-слой (этапы 6–7), а он сейчас в конце.
- **DeepSeek/КНР — не только заметка, а риск листинга.** Отправка счетов/реквизитов в КНР-инференс
  несовместима с 152-ФЗ/коммерческой тайной части покупателей Маркета. Провайдер-агностичность
  (`ANTHROPIC_*`) — правильная страховка; **до публикации листинга (этап 4) согласовать с владельцем**,
  дефолтный провайдер для публичной версии (DeepSeek vs не-КНР). **Листинг не публиковать, пока один
  реальный портал не проведёт реальный импорт.**
- **worker/mcp — отдельные контейнеры раньше «под нагрузкой».** agent-run спавнит тяжёлый подпроцесс
  (OCR/Claude Code) — не держать в одном процессе с HTTP-backend дольше необходимого (Dockerfile split уже заложен).
- **Блокеры (`⛔`):** доступ к живому порталу для валидации НДС/OAuth (частично снят — есть `B24_HOOK`);
  юр-согласие клиента на облачный LLM.
