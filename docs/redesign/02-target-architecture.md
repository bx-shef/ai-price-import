# Целевая архитектура (редизайн procure-ai)

> Last reviewed: 2026-07-22

Как должно быть после редизайна. Синтез двух референсов: раскладка/дисциплина/лендинг/деплой —
из эталона `client-bank-alfa-by` (облачное приложение Маркета Б24); AI-слой извлечения — из
методологии репозитория `ai-agent` (`docs/09-tz/00_intake-ai/`, `docs/15-integrations/`). **NB:** исходный
замысел «изолированный MCP + CLI-агент» **не реализован** — фактически извлечение это tool-less
OpenAI-совместимый chat-вызов (DeepSeek/BitrixGPT), а поиск/запись в CRM — детерминированно в `crm-sync`.
Снимок «как есть» — [`00-legacy-architecture.md`](00-legacy-architecture.md); процесс/статусы —
[`01-project-map.md`](01-project-map.md).

**Ключевые решения (зафиксированы):** продукт — **облачное приложение для Маркета Bitrix24**
(мультитенант, OAuth на портал); **никакого кода для ядра/коробки Б24** — только MCP поверх
**стандартного REST API**; провайдер LLM — **BitrixGPT по умолчанию** (Bitrix AI Router — уход от прямого КНР-инференса, ⚠ владельцу сверить условия маршрутизации/юрисдикции с Битрикс; DeepSeek/custom — опцией, провайдер-агностично через `LLM_PROVIDER`); публичный лендинг — да.

## 1. Принципы

**Что импортируем и суть действия.** Вход — **любой документ с табличной частью** (накладная, счёт,
КП, прайс, спецификация — не привязываемся к «счёту»). Суть: **найти контрагента и внести товарные
позиции** в целевую CRM-сущность. Тип цели и её направление определяются правилами роутинга
(единый режим или по типу/ключевым словам документа) — см. принцип 5.

1. **Единое Nuxt-приложение (монолит).** Лендинг, in-portal UI и Nitro-backend — в одном репо
   и одном приложении, как `client-bank-alfa-by`. Один домен, nginx проксирует `/api/*` в backend.
2. **Облачное приложение Маркета, мультитенант.** Каждый портал ставит приложение → событие
   `ONAPPINSTALL` приносит OAuth-креды; токены хранятся per-portal (Postgres, refresh шифруется).
   Никакого «одного целевого портала через вебхук» — портал определяется токеном установившего.
3. **Только стандартный REST, ноль кода в ядре Б24.** Убираем PHP-модуль коробки и патчи ядра.
   Поиск/создание сущностей — штатными методами (`crm.*`, `catalog.*`) по OAuth-токену портала.
4. **Изолированный MCP (обязателен для AI-проектов).** Агент работает с **абстрактными**
   инструментами (`find_supplier`, `find_product`, `create_target`) и не знает, что за ними физически
   (Bitrix24 REST сейчас, 1С — потом подменой транспорта). Backend не ходит в Bitrix24 REST из
   бизнес-логики напрямую — только через MCP. **Поиск договора в стандартное приложение не входит**
   (решение Q8) — это платная индивидуальная доработка на сервере клиента.
5. **Пер-портальная настройка маппинга.** Приложение generic → нельзя хардкодить инфоблоки/поля
   конкретного клиента. Настройки портала (`app.option` через server-side REST) задают: каталог(и),
   **поле артикула поставщика** (+ его тип и разделитель, см. ниже), **целевую сущность и её
   направление** (см. ниже), **тумблер «сохранять файл или нет»** (само дело —
   настраиваемое, `layout` задаёт приложение, админ его НЕ настраивает), **стратегию поиска товара**
   (по названию целиком / по артикулу), **стратегию при отсутствии товара** (создать товар с артикулом /
   пропустить строку+предупреждение / произвольная позиция), **настройки единиц измерения**
   (словарь: единица документа → `catalog.measure` + дефолтная единица + при отсутствии авто-создать),
   **чат ошибок** (`dialogId`) и чат уведомлений. **НДС** — ставки берутся из настроек Б24
   (`crm.vat.list`), см. «Модель НДС» ниже (в маппинг ставки не дублируем). Чистое ядро маппинга — в `app/utils` с тестами (как матрицы эталона). **Договор в маппинг не входит** (Q8) — платная доработка. **Налоговый ID
   поставщика в маппинг НЕ входит** — он во всех странах в одном поле `RQ_INN` (см. «Поиск поставщика»).

   **Целевая сущность импорта — расширяемая (не только сделка).** Через универсальный `crm.item.add`
   цель может быть: **сделка** (`entityTypeId=2`), **лид** (`entityTypeId=1`, #135 — с уточнением по
   контрагенту, см. §5 «Лид как цель»), **смарт-процесс** (динамический, `entityTypeId≥1000`) или
   **смарт-счёт / инвойс** (`entityTypeId=31`). У сделки/смарт-процесса/инвойса — своё **направление**
   (воронка) через `crm.category.*` и стадия; **у лида воронок нет** (`crm.category.list` для etid=1 →
   `ENTITY_TYPE_NOT_SUPPORTED`, live) — только опц. стадия. Позиции — универсальным
   `crm.item.productrow.set`. **КП / предложение (`entityTypeId=7`) целью НЕ является** (#135): (а) нет
   фильтруемого маркера идемпотентности (`originId`/`xmlId` отсутствуют), (б) по смыслу КП — исходящий
   документ *компании клиенту*, а на входе у нас документ *от контрагента*, импортировать его в КП нечего.
   **Роутинг цели — структура.** Цель = `entityTypeId` (сделка/лид/смарт-процесс/инвойс; КП убран — #135) +
   `categoryId` (направление; у лида не применяется) + стадия. Определяется по **трём уровням приоритета**:

   1. **Ручной override у файла** — менеджер указал цель при загрузке. Всегда побеждает.
   2. **Правила роутинга** (упорядоченный список из настроек) — применяется **первое сработавшее**.
   3. **Дефолтная цель** — если ни одно правило не сработало. Обязательна (fallback).

   **Fallback при удалённом направлении (устойчивость к рассинхрону настроек и портала).** Направление
   (`categoryId`) в правиле/дефолте может ссылаться на воронку, которую **удалили в CRM, а настройки не
   поправили** — тогда `crm.item.add` отверг бы цель. Перед созданием crm-sync **проверяет валидность
   направления** (`crm.category.list` по типу сущности) и, если воронка исчезла, спускается по цепочке:
   - **правило/ручной override с удалённым направлением → дефолтная цель;**
   - **дефолтная цель с удалённым направлением → жёсткий якорь `FALLBACK_TARGET` = сделка
     (`entityTypeId 2`), направление 0 (Default pipeline)** — всегда существует.

   Цель **без `categoryId`** (просто «сделка») валидна как есть — сущность создаётся в своей дефолтной
   воронке. **Якорь `FALLBACK_TARGET` (сделка/0) не проверяется REST-ом** (направление 0 существует
   всегда) — поэтому обычный импорт по дефолтной цели (она и есть `FALLBACK_TARGET`) **не делает лишних
   вызовов**; `crm.category.list` идёт только когда цель пиннит **не-дефолтное** направление. Список
   воронок мемоизируется по типу сущности (один вызов на тип). Проверка **fail-open**: если список
   воронок не удалось прочитать, берём цель как есть (не блокируем импорт; редкий персистентный сбой
   чтения категорий деградирует к обычному «create отвергнут → ретрай»). Валидируется **только
   `categoryId`** (не `stageId`). Тот же `FALLBACK_TARGET` — фолбэк парсера настроек, когда дефолтная цель
   в `app.option` пустая/битая (сделка, 0-е направление). **Спуск фиксируется предупреждением** в сводке
   прогона (не молча) — оператор видит, что документ ушёл в запасную цель.

   **Правило = `{ match, target }`.** `match` — условие одного из типов (можно комбинировать):
   - **`type`** — тип документа, который **классифицирует агент** (накладная / счёт / КП / прайс /
     спецификация / …), язык-агностично;
   - **`keyword`** — вхождение слова/фразы в текст документа (напр. «ТН»), проверка
     **детерминированная** (нормализация + homoglyph-fold).

   `target` — та же тройка `entityTypeId`+`categoryId`+`stageId`. **«Всё в одну цель»** — частный
   случай: список правил пуст → всё идёт в дефолтную цель (отдельный «режим» не нужен).

   **Направление (воронка/`categoryId`) выбирается в UI** (не только через `app.option`): и у дефолтной
   цели, и у каждого правила рядом с типом сущности есть пикер направления — список тянется с портала
   (`crm.category.list` по `entityTypeId`, фрейм-токеном: роут `GET /api/crm-categories`, чистое ядро
   `fetchCrmCategories`, admin-гейт). У лида воронок нет → пикер скрыт. Смена типа сущности сбрасывает
   `categoryId`, невалидный для нового типа. **Стадия (`stageId`) — тоже выбирается в UI** (каскад от
   направления): пикер тянет `crm.status.list` с портала (`fetchCrmStages`/`GET /api/crm-stages`, admin,
   чистое ядро `stagePicker.ts`), `ENTITY_ID` зависит от типа+воронки (live-verified: `DEAL_STAGE`/
   `DEAL_STAGE_<N>` у сделки, `SMART_INVOICE_STAGE_<cat>` у смарт-счёта, `DYNAMIC_<etid>_STAGE_<cat>` у
   смарт-процесса). **У лида стадию не выбираем** (`stageEntityId(1)→null`, пикер скрыт, `createTargetItem`
   не шлёт `stageId`): live-verified — `crm.item.add` для лида **молча игнорирует** `stageId`/`statusId`
   (лид садится на портальную дефолтную стадию, без ошибки), так что пикер был бы контролом-пустышкой
   (симметрично carve-out'у `categoryId` лида, #135). Пусто ⇒ дефолтная/первая стадия; смена направления/
   типа сбрасывает `stageId`; **невалидный/удалённый `stageId` для сделки/смарт-* Б24 тоже игнорирует
   молча** (создаёт в дефолтной стадии, без reject → нет retry-loop), поэтому `resolveValidTarget`
   валидирует только `categoryId` (у него невалидное значение реально reject'ит). Есть и у дефолт-цели, и у
   каждого правила, и в ручном выборе при загрузке на `/app`.
   **Ручной выбор цели при загрузке (`manualOverride`) — сделан end-to-end:** на
   `/app` свёрнутый блок «Куда импортировать» (Авто/Лид/Сделка/Смарт-счёт + пикер направления + стадии) → поле `target`
   (JSON) в `POST /api/import/upload` → чистый валидатор `parseManualTarget` (недоверенный вход) → поле
   `manualOverride` Redis-задания (`jobStore`, #B/#D — не Postgres) → `getManualOverride` в agent-run → `RoutingSignals.manualOverride`
   → `resolveTarget` применяет с высшим приоритетом. «Авто» (пусто) ⇒ идём по правилам. Сервер
   ре-валидирует сохранённый target (tamper-safe).

   **Резолвинг — чистая функция** `resolveTarget({ documentType, text, manualOverride }, rules,
   defaultTarget)` в `app/utils`, покрыта тестами (паттерн матриц/аллокатора эталона). Классификацию
   `documentType` даёт агент; матч ключевых слов и применение правил по порядку — детерминированно.

   Примеры конфигураций:
   - *Единый:* правил нет; дефолт = Сделка / направление «Закупки».
   - *По типу:* `[счёт→инвойс]`, `[КП→КП]`; дефолт = Сделка / «Закупки».
   - *По ключевику:* `[text⊃«ТН» → Сделка / направление «Поставки»]`; дефолт = инвойс.

   **Артикул поставщика (термин верный).** Один товар каталога может нести **несколько** артикулов
   поставщика в одном поле. Поле бывает двух типов, админ указывает тип в настройках:
   - **тип «текст» (multiline)** → каждый артикул **с новой строки**;
   - **тип «строка» (string)** → артикулы **через разделитель**; **разделитель админ задаёт сам**
     (в поле нет «правильного» дефолта — например `,`, `;`, `/`).

   Чистая функция `parseSupplierArticles(fieldValue, {kind, delimiter})` → нормализованный набор
   артикулов (trim, отбросить пустые, homoglyph-fold). Матчинг товара сверяет извлечённый из
   документа артикул с этим набором.
6. **Чистое ядро в `app/utils`, покрытое тестами.** Детерминированная логика (классификация,
   нормализация, homoglyph-folding, сборка payload сделки, распознавание идентификаторов) — чистые
   функции без I/O, юнит-тестируемые без Nuxt/БД/сети. I/O и реактивность — на краях.
7. **Тяжёлое — через очередь.** Извлечение текста, прогон агента, запись в CRM — джобы BullMQ+Redis,
   идемпотентные, с ретраями. Backend не держит состояние в памяти → масштабируется.
8. **Server-side REST по OAuth-токену портала.** Фрейм-SDK Б24 — только установка и UI-хром
   (`setTitle`/`fitWindow`). Данные/настройки — серверным REST по сохранённому токену.
9. **Безопасность.** Least-privilege allowlist агента, framing недоверенного документа, non-root
   контейнеры, magic-byte MIME, SSRF-allowlist, секреты вне argv/логов, шифрование refresh в покое.

## 2. Схема

```
Браузер (лендинг / B24 iframe) ──HTTPS──▶ nginx (:8080, unprivileged, CSP-hashing)
        │                                     │  /  → статика Nuxt (лендинг + in-portal UI)
        │                                     │  /api/* → backend:3000 (Nitro)
        ▼
  ┌──────────────────────────────────────────────────────────────────────────┐
  │ app (nginx + статика Nuxt generate)   │  backend (Nitro node-server)       │
  │  лендинг /                            │   server/api/* (upload, job, health,│
  │  in-portal /app /import /install ...  │     b24/events, queues, settings)   │
  │                                       │   server/queue/ (BullMQ producers)  │
  │                                       │   server/utils/ (pure, DI, tests)   │
  └───────────────────────────────────────────────┬───────────┬───────────────┘
                                                   │           │
                                          enqueue  │           │  writes
                                                   ▼           ▼
  ┌──────────────────────────┐        ┌────────────────────┐  ┌──────────────┐
  │ worker (BullMQ)          │        │ Redis (очереди)    │  │ Postgres     │
  │  file-extract → текст    │◀──────▶│ file-extract       │  │ OAuth-токены │
  │  agent-run   → Claude ───┼──MCP──▶│ agent-run          │  │ per-portal,  │
  │  crm-sync    → запись    │        │ crm-sync           │  │ идемпот.,мет.│
  └──────────┬───────────────┘        └────────────────────┘  └──────────────┘
             │ MCP (Streamable HTTP, Bearer)
             ▼
     ┌────────────────────────────────────────────────┐
     │ MCP-сервер (изолированный, мультитенант)        │
     │  find_supplier / find_product(s) /              │
     │  create_target (deal/smart/invoice/quote)       │
     │  └─ СТАНДАРТНЫЙ REST по OAuth-токену портала ───┼──▶ Bitrix24 (облако/коробка клиента)
     │     crm.* / catalog.* — без кода в ядре Б24     │    crm.company.list, crm.requisite.*,
     └────────────────────────────────────────────────┘    crm.product.list, crm.item.add,
                                                            crm.category.*, crm.item.productrow.set, ...
```

Границы деплоя: `app` (статика+nginx), `backend` (Nitro), `worker` (в MVP совпадает с backend,
выносится под нагрузкой), `mcp` (наружу не публикуется), `redis`, `db` (Postgres). **PHP-модуля
коробки и патчей ядра больше нет.** Bitrix24 клиента — внешняя система, доступ только по OAuth-REST.

```
                          #   (create-target: crm.item.add по entityTypeId+categoryId; deal/smart/invoice/quote)

```

**Каталога `b24-controller/` (PHP) в целевой раскладке нет** — вся интеграция ушла в MCP поверх
стандартного REST. Философия split'а (из эталона): `utils` — чистое; `composables` — реактивное;
`config` — данные; `types` — типы; `server/utils` + `server/queue/handlers` — чистые обработчики с DI
(`HandlerDeps`), живая проводка в `worker.ts`, фейки в тестах.

## 4. Потоки данных

**Импорт файла (happy path):**
1. `POST /api/import/upload` → сохранить файл, создать задание (Redis, TTL — не Postgres, #B/#D), **опц. ручной override цели**
   рядом с файлом (тип сущности + направление), `enqueue file-extract`.
2. `file-extract` (worker): pdftotext / OCR (tesseract `rus`+`bel`+`kaz`+`eng`) / office → `DOCUMENT_TEXT` → `enqueue agent-run`. Языки документов: рус/бел/каз — см. [`06-multilingual.md`](06-multilingual.md).
3. `agent-run` (worker): OpenAI-совместимый chat-вызов (DeepSeek/BitrixGPT, in-process) с промптом +
   `DOCUMENT_TEXT` → извлечённая структура позиций (чистый экстрактор, **без** доступа к Bitrix24);
   поиск поставщика/товаров и запись — детерминированно в `crm-sync`. Результат → `enqueue crm-sync`.
4. `crm-sync` (worker): **резолв цели** (ручной override → авто-роутинг
   агента → дефолт из настроек) → `create_target` через MCP → **стандартный REST** создаёт цель
   (`crm.item.add` по `entityTypeId`+`categoryId`; для сделки/КП/смарт-счёта — с явными
   `opportunity`+`isManualOpportunity`), пишет позиции (`crm.item.productrow.set`, штатный
   НДС), кладёт файл на Диск (папка → подпапка по месяцам) и создаёт **дело** с файлом
   (без коммента/UF-поля); счётчики метрик.
5. UI поллит `GET /api/import/status` → результат по файлу: статус (создано / без сделки + причина) +
   **ссылка «открыть»** созданную сущность в портале (по типу: `crm.item` detail-путь;
   в портале — через `BX24.openPath`/слайдер, работает и в вебе, и в мобильном приложении).

**Импорт доступен только внутри портала Б24 (in-portal, iframe) — веб И мобильное приложение.**
Отдельной standalone-загрузки (вне портала) не делаем — это кастомная доработка (в маркетинг как
платная фича). Публичным остаётся только **лендинг** (`/`); рабочая страница импорта — **`/app`**
(`/import` слит в неё и теперь редиректит), требует фрейм-контекст (`useB24().init()` no-op вне фрейма → приглашение открыть в портале).

**Мобильное приложение Б24.** Всё должно работать в мобильном приложении Битрикс24: адаптивный UI
(`b24ui` mobile-first), приложение открывается своей страницей из левого меню (`placement.bind` не
используем), открытие сущности/слайдера и загрузка файла (камера/файлы телефона) внутри мобильного вебвью.
Проверяем на реальном мобильном клиенте (визуальная верификация).

**Очередь — обязательна для всего конвейера (по образцу эталона `client-bank-alfa-by`).**
`POST /api/import/upload` **не обрабатывает синхронно** — только кладёт файл, создаёт задачу и
`enqueue`; ответ возвращается сразу, вся работа идёт в worker'ах через BullMQ+Redis. Переносим
из эталона проверенные паттерны 1-в-1:
- `server/queue/topology.ts` — **чистые контракты**: очереди (`b24-events`, `file-extract`,
  `agent-run`, `crm-sync`), payload'ы джоб, **детерминированные идемпотентные `*JobId`** (дедуп
  ретраев; разделитель `|`, не `:`).
- `connection.ts` — ленивый `getQueue`, опции из `REDIS_URL` (без прямой зависимости от ioredis);
  дефолты `attempts: 3`, exp-backoff 5s, `removeOnComplete/Fail`; гуард `queueEnabled()`.
- `producers.ts` — `enqueue*` (**no-op без Redis**).
- `handlers.ts` — **чистые обработчики с DI** (`HandlerDeps`), тестируются на фейках; живая проводка
  транспортов (extract/agent/REST) — в `worker.ts`.
- Для не-ретраящихся событий Б24 (`/api/b24/events`) — **синхронный фолбэк в БД**, если Redis недоступен
  (как в эталоне), чтобы установка/удаление не потерялись.

В MVP worker крутится **в процессе backend** (решение D2); под нагрузкой выносится в отдельный контейнер.

**Установка в портал (мультитенант):** `/install` → `init → app.info/scope/event.get →
event.bind(ONAPPINSTALL/ONAPPUNINSTALL → /api/b24/events) → installFinish`. Событие `ONAPPINSTALL`
на `/api/b24/events` приносит OAuth-креды → токены в Postgres (refresh шифруется). UF-поля/каталог —
не создаём молча: показываем чек-лист настройки и форму маппинга (`/settings`).

**События Б24:** `POST /api/b24/events` → верификация `application_token` (fail-closed) → очередь
`b24-events`; consumer — единственный писатель токенов. `ONAPPUNINSTALL` → чистим токены/данные портала.

**Продление токенов OAuth (две дорожки).** Официальная модель Б24
([auto-renewal](https://apidocs.bitrix24.ru/api-reference/oauth/auto-renewal.html)): `access_token`
живёт **1 час**, `refresh_token` — **180 дней**. Б24 **не рекомендует** обновлять по короткому
расписанию (раз в час/сутки) — лишняя нагрузка и риск авто-блокировки приложения.
1. **Ленивый рефреш (основной путь).** `ensureAccessToken`: используем сохранённый `access_token`,
   при ошибке протухания — один запрос `grant_type=refresh_token` (client_id/secret из env),
   сохраняем новую пару, повторяем REST. Так работает любой активный портал.
2. **Проактивный keep-alive (крон, для простаивающих порталов).** Наше приложение upload-driven:
   портал может месяцами не грузить файлы → без активности `refresh_token` протухнет на 180-й день
   и **портал будет потерян**. Поэтому крон **раз в сутки** делает дешёвый `SELECT` порталов, у
   которых `refresh_token` близок к 180-дневному истечению (порог **~177 дней** без обновления =
   осталось <3 дней), берёт client-секреты из env и обновляет **только их**. Это НЕ «каждый час» —
   обновляем лишь то, что вот-вот протухнет (соответствует рекомендации Б24).

Для этого в `portal_tokens` храним `refreshed_at` (когда последний раз получили пару) и/или
`refresh_expires_at` (= `refreshed_at + 180d`). Ошибка продления (`PAYMENT_REQUIRED`, приложение
удалено, истёк период) → помечаем портал неактивным + метрика/алерт, не роняем крон. Крон живёт в
плагине `plugins/queue.ts` (как демо-крон эталона; отдельного `server/queue/cron.ts` нет); реализация — чистая `selectTokensNearExpiry`
+ DI-рефреш (юнит-тесты без сети).

**Запись в CRM (только стандартный REST через MCP):**
- Поставщик — по **налоговому идентификатору** через **единое поле реквизита `RQ_INN`**
  (`crm.requisite.list` фильтром `RQ_INN=<value>` → компания по `ENTITY_TYPE_ID`). Поле `RQ_INN`
  **одинаково для всех стран** — меняется лишь смысл: RU=ИНН, BY=УНП, KZ=БИН/ИИН и т.д. (шаблоны
  реквизитов пер-страна: `crm.requisite.preset.countries` → `ID/CODE/TITLE`, напр. RU=1, BY=4).
  Поэтому REST-поиск **страно-агностичен** — просто сверяем значение `RQ_INN`; вся страно-/язык-
  зависимость вынесена в **промпт агента** (см. ниже).
- Договор — **не ищем** (решение Q8). Подбор договора — платная индивидуальная доработка на
  сервере клиента (маркетинговая фича, см. `04-marketing-landing.md`).
- Товар — `crm.product.list` по **полю артикула поставщика** из маппинга. **Проверено вживую
  (2026-07-09, свойство создано на iblock 25):** exact-фильтр `PROPERTY_<id>` **НЕ находит** товар,
  если в поле несколько артикулов; работает только **подстрочный `%PROPERTY_<id>`** (LIKE). Поэтому
  lookup двухшаговый: `%PROPERTY_<id>` (order ID ASC) сужает на стороне портала → **точная сверка**
  на клиенте через `parseSupplierArticles`/`articleMatches` (отсекает ложные `STP-5`⊂`STP-50`).
  **Поле — только числовой `PROPERTY_<id>`** (символьный код `%LIKE` не фильтрует — вернул весь
  каталог; в форме настроек — пикер свойства даёт id). Работает для обоих вариантов; поле парсится по
  своему типу (текст = построчно / строка = через админский разделитель) в набор артикулов
  (`parseSupplierArticles`), сверка с извлечённым артикулом (homoglyph-tolerant); родительский товар.
- Целевая сущность — универсальный `crm.item.add` по `entityTypeId` (2=сделка / 1=лид, #135 / ≥1000=смарт-процесс /
  31=инвойс; КП/7 исключён — #135) + `categoryId` (направление из `crm.category.*`; у лида нет) + стадия. Позиции —
  `crm.item.productrow.set` (`ownerType` по типу: D/SI/`T<id>`; **лид — `L`**, live-verified).

  #### Лид как цель (#135)

  **Мотив (решение владельца):** КП импортировать нечего (исходящий документ компании), а вот **лид** —
  осмысленная цель: входящий документ от нового/существующего контрагента заводит лид для квалификации.
  Логика та же, что у сделки, **плюс уточнение по контрагенту**.

  **Согласованные решения:**
  - **Тип цели задаёт пользователь/админ, не авто-определяем.** Цель выбирает менеджер (override при
    загрузке) или админ (правила роутинга + дефолт). Наличие компании **НЕ** переключает тип: если
    сконфигурирован лид — всегда создаём лид (не «сделка при найденной компании»); found/not-found
    влияет только на `companyId` vs `companyTitle` внутри лида.
  - **Стадия — пустая** (портальная дефолтная NEW); отдельного поля стадии лида в настройках не даём.
  - **Из документа — только название контрагента** (`companyTitle`). Контактное лицо / телефон / e-mail /
    произвольные поля лида — **платная индивидуальная доработка на сервере клиента** (self-hosted, как и
    подбор договора Q8): «на вашем сервере сделаем что нужно». В generic-облаке — только название.

  **Живые факты (`crm.item.fields` etid=1, тестовый портал 2026-07-18):** лид несёт `title`, `companyId`
  (`crm_company`), `opportunity`+`isManualOpportunity`+`currencyId` (сумма как у сделки), `stageId`
  (`crm_status`, глобальные стадии лида — **воронок/`categoryId` у лида нет**), контрагентские поля
  `companyTitle`/`name`/`lastName`/`secondName`/`post`, маркер `originId`+`originatorId` (идемпотентность
  уже live-verified — как сделка). Позиции — `productrow.set` `ownerType='L'` (`T1` → `ACCESS_DENIED`, live).

  **Маппинг полей при создании:**

  | Поле лида (crm.item, etid=1) | Источник | Условие |
  |---|---|---|
  | `originId` / `originatorId` | `jobId` / originator | всегда (маркер идемпотентности) |
  | `title` | `Импорт: <поставщик>` (≤255) | всегда |
  | `opportunity` + `isManualOpportunity:'Y'` | Σ строк (gross, `computeOpportunity`) | если есть строки |
  | `currencyId` | `doc.currency` | если валюта задана |
  | `stageId` | `target.stageId` | опц. (пусто → дефолтная NEW-стадия) |
  | `companyId` | найденная компания (`RQ_INN`) | **только если поставщик НАЙДЕН** |
  | `companyTitle` | имя поставщика из документа (≤255) | **только если поставщик НЕ найден** |
  | product rows (`ownerType:'L'`) | позиции документа | всегда |

  **Уточнение по контрагенту (ключевое отличие от сделки):**
  - **Поставщик найден** (`findCompanyByTaxId` → `companyId`): ставим `companyId` — лид привязан к
    существующей компании («повторный лид»); контрагентские поля (`companyTitle`/`name`/…) **не заполняем**.
  - **Поставщик НЕ найден:** `companyId` не ставим, **заполняем `companyTitle`** = название из документа —
    менеджер квалифицирует «сырой» лид. Это **снимает UNMATCHED-тупик** для лид-цели: в отличие от сделки
    (где «не найден» = warning + лид/сделка без компании), лид всегда несёт имя контрагента.

  **Реализовано (#135):** (1) `TargetEntityKind` += `'lead'`, `ENTITY_TYPE_ID.lead=1`, `originStrategy(1)`
  уже `'origin'` — маркер готов; (2) `ownerTypeCode(1)` → `'L'`; (3) `supportsOpportunity(1)` → `true`;
  (4) `entityOpenPath(1)` → `/crm/lead/details/<id>/`; (5) в `runCrmSync` — ветка полей для лида
  (found→`companyId` / not-found→`companyTitle`); (6) форма настроек — пресет «Лид» (+ сброс `categoryId`
  при переключении на лид); (7) **гард `categoryId` у лида** — `createTargetItem` не шлёт `categoryId` при
  etid=1 (у лида нет категорий: `crm.item.add` иначе падает «Item has no CATEGORY_ID field», live-verified),
  защита от значения, унаследованного из dealrouting-правила; (8) тесты
  (found/not-found, non-lead без companyTitle, no-name, marker, ownerType, opportunity, categoryId-skip). **Live-verified на
  тест-портале** (`B24_HOOK`): `crm.item.add` etid=1 с маркером+`companyTitle`+`opportunity`+валютой →
  round-trip `crm.item.get` подтвердил персист; `productrow.set ownerType='L'` ОК; поиск по маркеру нашёл
  созданный лид; очистка `crm.item.delete` ОК.
- **Модель НДС.** Позиция несёт `taxRate` (ставка) и `taxIncluded` (цена с НДС / без):
  - **Ставки — только из настроек портала Б24** (`crm.vat.list`/`catalog.vat.list`); других ставок нет.
    Ставку из документа сопоставляем с доступными в портале. **Нет такой ставки** (напр. в документе
    25%, в портале только 20%) → **ошибка → чат ошибок**.
  - **`taxIncluded` — единый на весь документ** (определяет разбор документа/агент: всё «с НДС» ИЛИ
    всё «без»). **Разнобой в одном документе не допускается** → **ошибка → чат ошибок**.
  - **Ставки НДС могут различаться по товарным позициям** (per-line `taxRate` — норма).
- **Исходный файл и дело (вместо коммента/UF-поля).** Файл кладём **обязательно на общий Диск Б24**:
  общий диск (`disk.storage.getlist`) → **папка приложения** → подпапка **по месяцам** (`YYYY-MM`,
  `disk.folder.addsubfolder` идемпотентно) → `disk.folder.uploadfile`. Затем на целевой сущности
  создаём **настраиваемое дело** `crm.activity.configurable.add` (только в контексте приложения;
  `ownerTypeId`+`ownerId` = цель; `layout` **задаёт приложение** — иконка, ссылка «открыть»,
  краткие итоги; можно вшить кнопку/`openRestApp`). **Привязка файла к делу (#129 follow-up):** архив
  на Диске отдаёт `DETAIL_URL` (у `uploadfile` — **абсолютный** `https://<портал>/docs/file/…`,
  live-verified), сохраняется в поле `diskFile` Redis-задания (`jobStore`, #B/#D); при записи дела `crm-sync` читает его и
  добавляет кнопку **«Исходный файл»** в `footer.buttons` — URL нормализуется в **относительный** путь
  портала (`detailUrlToRelative`: редирект не уходит с портала; невалидный/протокол-относительный →
  кнопки нет). **Админ настраивает только «сохранять файл или нет»** (тумблер) — сам вид дела не настраивается. **Комментарий в таймлайн и UF-поле в сущности НЕ
  пишем** (уходим от модели старого проекта `UF_CRM_DEAL_SH_PRCHS_AI_FILE` + `COMMENTS`).
  - **Проверено вживую (тестовый портал, вебхук).** Общий диск = `disk.storage.getlist` с
    `ENTITY_TYPE='common'` (у нас ID 3, «Общий диск»); `disk.folder.addsubfolder` (папка `procure-ai-import`)
    и `disk.folder.uploadfile` (`fileContent: [name, base64]`) отрабатывают. `crm.activity.configurable.add`
    **через вебхук отдаёт `ERROR_WRONG_CONTEXT`** — метод жив только в OAuth-контексте приложения; наш
    `crm-sync` ходит per-portal OAuth-токеном (`makePortalSdkCall`, SDK-транспорт), поэтому в проде корректен, но
    вебхуком не проверяется. **Файловый фолбэк, проверяемый вебхуком:** `crm.activity.todo.add` с
    `fileTokens[0]=disk<ID>` прикрепляет диск-файл к делу (нужен `deadline`). Чат: `im.message.add`
    (`DIALOG_ID`/`MESSAGE`) — прогон round-trip + `im.message.delete` — ОК.
  - ⚠ **Жизненный цикл исходного файла.** Загруженные байты **удаляются сразу после file-extract**
    (`worker.ts` `cleanupUpload`) — к шагу `crm-sync` файла уже нет. Поэтому сохранение на Диск должно
    происходить **на стадии file-extract** (пока байты есть), а id диск-файла — переживать до `crm-sync`
    (в doc-store/записи джобы), чтобы дело сослалось на него. Это отдельный микрослайс плюмбинга.

**Промпт агента: извлечение налогового ID с учётом страны и языка.** Поиск по `RQ_INN` страно-
агностичен, но **распознать** идентификатор в документе — задача агента, и она страно-/язык-зависима:
- В промпте — краткий справочник терминов налогового ID по странам ЕАЭС/СНГ: RU **ИНН** (10/12 цифр),
  BY **УНП** (9 цифр), KZ **БИН**/**ИИН** (12 цифр), и т.д. — как метку идентификатора и как формат.
- **Мультиязычность:** документ может быть на русском, белорусском, казахском и др. — агент (DeepSeek)
  распознаёт метку идентификатора **на языке документа** (напр. каз. «БСН/ЖСН») и извлекает число.
  Отдельный OCR/перевод не нужен — LLM работает с текстом на любом языке.
- Извлечённое значение агент кладёт в `find_supplier` → REST-поиск по `RQ_INN`. Нормализация числа
  (убрать пробелы/разделители) — чистая функция, покрыта тестами.
- Если налоговый ID не распознан/не найден — путь «поставщик не найден» (см. вопросы процесса).

### Решения по проводке crm-sync (из ревью)

- **Где резолв.** Агент делает **извлечение и классификацию** (поставщик-taxId, позиции, артикулы, тип
  документа). **Детерминированный поиск** компании (по `RQ_INN`) и товара (по артикулу) — в шаге
  `crm-sync` через MCP-инструменты (`find_supplier`/`find_product`), а не LLM: детерминированно,
  тестируемо, без траты токенов и галлюцинаций. Инструменты MCP остаются единственной дверью в Б24.
- **Без потерь строк.** Жёсткая нестыковка (ставка НДС/валюта не в портале) **не создаёт частичную
  сущность и не роняет строки** — весь документ уходит в чат ошибок; оператор правит портал и
  переимпортирует (1-в-1). Мягкая (единица не сопоставлена → дефолт) — предупреждение, создаём.
- **Идемпотентность.** Созданная сущность несёт **маркер job-id** (`originId`+`originatorId` у
  сделки/лида/контакта/компании; `xmlId`=`<originator>:<jobId>` у инвойса/смарт-процессов —
  live-verified `crm.item.fields`). Перед созданием `crm-sync` **ищет этот маркер в Б24**
  (`crm.item.list filter[originId]`/`filter[xmlId]`) — нашёл ⇒ **до-записывает** позиции
  (`productrow.set` — полная замена), не нашёл ⇒ создаёт с маркером. **Источник истины — портал, не
  локальная БД**: это закрывает старое окно create→checkpoint (даже если ретрай стартует после
  создания, но до записи чекпоинта — маркер на сущности находится). Таблица `job_result` **упразднена**
  (#135, `DROP TABLE IF EXISTS` в `schema.ts` — всегда была пустой, мигрировать нечего). Цена — +1 `crm.item.list` на создание.
  Тип без фильтруемого маркера (напр. КП/7) целью **не является** — его убрали (#135).
- **Модель `application_token`** (B24 [«Безопасность в обработчиках»](https://apidocs.bitrix24.ru/api-reference/events/safe-event-handlers.html)):
  `application_token` — **не пред-общий секрет**, он приходит в первом `ONAPPINSTALL` и запоминается
  по порталу. Известный портал: событие сверяется с **сохранённым** токеном (единственный способ для
  `ONAPPUNINSTALL` — в нём нет auth-данных). Первая установка: токена ещё нет — подлинность проверяется
  по доставленному `access_token` (дешёвый authed-вызов `profile`, `server/utils/verifyInstallToken`;
  подделать рабочий OAuth-токен нельзя). env `B24_APPLICATION_TOKEN` — **опциональный** доп. глобальный
  гейт первой установки (обычно пуст). Ротация токена (`OnAppUpdate`) — при живой доработке (сейчас write-once).

### Роль агента: чистый экстрактор (MVP) vs драйвер MCP (будущее)

> **Зафиксировано (реализация).** В MVP `agent-run` агент — **чистый экстрактор**: получает
> **только текст документа** на stdin, возвращает JSON `ExtractedDocument`, **без доступа к
> Bitrix24** (все встроенные инструменты запрещены, MCP-сервер не подключается —
> `buildAgentArgs()` без `--mcp-config`). Это прямое следствие решения «резолв — в `crm-sync`,
> **а не LLM**»: недоверенный документ, попав в модель, может только вернуть JSON — он не может
> тронуть портал или что-то выгрузить. Детерминированный поиск поставщика/товара и запись цели
> выполняет **`crm-sync`**, вызывая тела абстрактных инструментов (`find_supplier`/`find_product`/
> `create_target`) **в процессе** (единственная дверь в Б24 сохранена на уровне `crm-sync`).
> Реализация: `server/agent/chatExtract.ts` (+ нормализатор `app/utils/extractedDocument.ts`).
>
> HTTP-путь «агент → MCP по Bearer» ниже — **для будущего режима**, где LLM сам обогащает данные
> через инструменты; `buildMcpConfig`/`agentAllowedTools` заложены под него, но MVP их не задействует.

### MCP: авторизация и токены (агент/DeepSeek НЕ видит токены доступа)

Принцип: **LLM-агент видит только интерфейс MCP-инструментов, но не токены доступа к Bitrix24.**
Токены передаются в MCP **через файл**, а не через контекст модели (как в старом проекте файл
документа передавался вне LLM). Схема на джобу `agent-run` (**будущий** режим-драйвер):

1. Backend пишет **временный конфиг-файл** (`0600`, вне argv/`ps`) с: URL MCP-сервера + **короткоживущий
   per-job bearer к самому MCP** + идентификатор джобы/портала (`member_id`). Этот bearer — авторизация
   агента к MCP, НЕ токен Bitrix24.
2. Агент (LLM) запускается с этим MCP-конфигом; в его контекст токен Bitrix24 **не
   попадает никогда**.
3. MCP при вызове инструмента по `member_id` берёт **OAuth-токен портала** из общего защищённого
   источника (backend/Postgres, refresh расшифровывается), делает `ensureAccessToken` и зовёт REST.
   Токен доступа живёт на стороне MCP/backend, не в LLM.
4. Файл документа тоже вне LLM: инструменты читают файл по пути (backend-скоуп) и работают с ним
   (base64/Диск) — бинарник в контекст модели не уходит.

Итог: даже если провайдер LLM — внешний (DeepSeek/КНР), ни OAuth-токен портала, ни бинарные файлы
через него не проходят; наружу уходит только **текст документа** (см. `05-data-policy.md`). Доработка
MCP (порт из `legacy/mcp`+`legacy/mcp-overlay` → первоклассный пакет `mcp/`, команды на стандартный
REST, эта модель авторизации) — этап 6.

**Требования безопасности к MCP (при реализации, из ревью):**
- **Строгая привязка per-job bearer к `member_id`/джобе** и короткий TTL: MCP обязан отвергать
  вызовы, где `member_id` в аргументах не совпадает с привязанным к bearer (иначе «спутавшийся»
  агент вытянет данные чужого портала). Тенант-изоляцию проверяет MCP в коде, а не доверяет вводу агента.
- **Временный конфиг-файл** — в непубличной директории, удаляется после джобы (в т.ч. при сбое/краше).
- **Не логировать** полный `DOCUMENT_TEXT` и секреты в общие логи / payload'ы очередей / Postgres.
- **SSRF-гард** (`isSafeClientEndpoint`, порт из эталона) для любых URL из портала (client_endpoint,
  disk); **нейтрализация BB-кодов** (`neutralizeBb`) для текста из документа перед `im.message.add`.

### Матрица нестыковок (обработка исключений)

Различаем **предупреждение** (не блокирует, копится в отчёт по файлу) и **ошибку** (шлём в
**чат ошибок** портала — отдельный `dialogId` в настройках, как в эталоне). Правила:

| Нестыковка | Что делаем |
|---|---|
| **Поставщик не найден** (по `RQ_INN`) | Целевую сущность **всё равно создаём** (сделка/инвойс/смарт-процесс — по настройкам), без привязки компании. Предупреждение в отчёт. |
| **Товар не найден** | Зависит от настроек. **Что ищем** (настройка): по названию целиком / по артикулу. **Что делаем, если нет** (настройка, один из трёх): (а) **создать товар** с артикулом; (б) **не вносить строку** + предупреждение; (в) внести **произвольную товарную позицию** (free-form, без привязки к каталогу). |
| **Валюта** | Создаём сущность **в валюте документа** (`crm.item.add` принимает любую валюту — мы только создаём, не обновляем, **конвертации нет**). Если такой валюты **нет в списке валют портала** → **ошибка → чат ошибок.** |
| **Единица измерения** | Сопоставление по **словарю** (единица документа → `catalog.measure`); нет соответствия → дефолтная единица / авто-создать; при этом фиксируем как **ошибку → чат ошибок.** |
| **Ставка НДС не в списке портала** | Ставка из документа отсутствует в `crm.vat.list` (напр. 25% при доступном 20%) → **ошибка → чат ошибок.** |
| **Разнобой НДС в документе** | Часть позиций «с НДС», часть «без» (смешанный `taxIncluded`) → **ошибка → чат ошибок** (режим единый на документ). Разные `taxRate` по позициям — норма, НЕ ошибка. |
| **Несколько совпадений** (поставщик по `RQ_INN` не уникален; товар по артикулу/названию) | Детерминированный выбор **минимального ID** + предупреждение в отчёт. |
| **Дубликат** (повторная загрузка того же счёта) | **Дедупа нет — сущность создаём всегда** (решение владельца). |

Предпросмотр позиций до создания **не показываем** — создаём сразу, менеджер видит результат
(создано / без сделки + причина) и **ссылку «открыть»**. Некорректные цена/количество
(не число/отриц.) — клампим/помечаем предупреждением.

> **Идемпотентность ретрая (тех. нюанс, не бизнес-дедуп).** Раз бизнес-дедупа нет, при ретрае
> джобы `crm-sync` важно не создать сущность дважды: **маркер job-id на самой сущности**
> (`originId`/`xmlId`) + **поиск в Б24 перед созданием** (см. «Идемпотентность» выше) — повтор
> подхватывает найденную сущность. Источник истины — портал (без локального чекпоинта). Это защита
> от двойной записи при сбое, а не защита от повторной загрузки того же счёта пользователем (её нет).

## 7. Метрики, обратная связь, наблюдаемость, оператор

Всё скоуплено per-portal (мультитенант), хранение — Postgres; чистый расчёт презентации — `app/utils` с тестами.

### 7.1. Метрики — два уровня

- **Уровень 1 — всегда на виду (важные и мотивирующие).** Прямо на странице приложения (`/app`):
  обработано документов, создано сущностей, **сэкономлено времени/денег**, % успешных (создано vs
  без сделки). Компактно, «за сегодня / за всё время». Цель — показывать ценность каждый раз.
- **Уровень 2 — отдельная страница `/metrics`.** Детальные: форматы (PDF/скан/XLSX/DOCX),
  **match-rate** (поставщик/товар найден/нет), время по стадиям (extract/agent/crm-sync), кол-во
  позиций, AI-метрики (токены/латентность/ретраи/выбор инструментов), очереди.

### 7.2. Обратная связь (переносим из старого проекта)

- **Отзывы сотрудников** — на результате импорта: 👍 / 👎 / 💡(идея) + опц. текст. Хранятся
  (Postgres, per-portal) и **сразу уходят в закрытый (приватный) репозиторий издателя** через durable
  outbox (issues) — так издатель их **смотрит и разбирает** (как в старом проекте). Канал — приватный.
- **Согласие на отправку файла.** При отправке отзыва спрашиваем сотрудника: **прикладывать
  исходный файл документа или нет** (документ = коммерческие данные клиента). По умолчанию — **без
  файла**; файл уходит только с явного согласия. Этот момент фиксируется в **политике данных**
  ([`05-data-policy.md`](05-data-policy.md)).
- **Самооценка агента (сигналы)** — агент в выводе отдаёт сигналы уверенности/проблем (низкая
  уверенность в артикуле, не распознал УНП, спорная позиция) → телеметрия + маркеры «на ручной
  разбор» + алерты дрейфа.

### 7.3. Наблюдаемость (технический слой)

- `GET /api/health` — liveness (`{status,time,commit}`), docker healthcheck.
- Счётчики очередей BullMQ (`waiting/active/completed/failed/delayed`) → `server/queue/stats.ts` → рендер b24ui/HTML-карточками (графической библиотеки нет; `echarts` не подключён).

### 7.4. Страница оператора (как в эталоне)

Служебная зона за **авторизацией оператора** (HMAC-cookie сессия, `/login`, rate-limit — модель
эталона `client-bank-alfa-by`), отдельно от in-portal-страниц:
- **`/queues`** — живой мониторинг очередей (b24ui/HTML-карточки, без графической библиотеки); эндпоинты `/api/queues` (токен-заголовок,
  nginx `deny all`) и `/api/ops/queues` (сессия оператора).

Отдельной «очереди ручного разбора» **нет** (решение Q13): все исходы определены (предупреждения в
отчёте по файлу + чат ошибок), зависших на ручной разбор кейсов модель не создаёт.

### 7.5. Использование и политика

Алерты на всплеск `failed`/ошибок провайдера/падение match-rate; экономический дашборд — аргумент
ценности в маркетинге. **Дефолты видимости диагностики** — в отдельной политике (как
`DIAGNOSTICS_POLICY.md` старого проекта), без ad-hoc. Глубокая телеметрия (**OTel → ClickHouse →
Grafana**, методология `ai-agent` `docs/19`) — после MVP.

## 8. Требование точности: 1-в-1 со счётом

Позиции и цены в созданной сделке должны выходить **1-в-1 со счётом/накладной поставщика** —
без потерь строк, без искажения цен/количеств. Со штатным `crm.item.productrow.set` (без патча
ядра) бумажная цена мапится напрямую; НДС считается порталом штатно. **✅ Проверено вживую**
(`B24_HOOK`): `price=100, qty=2, taxRate=22, taxIncluded=Y` → `priceBrutto=100`, Итого=200.00,
НДС=36.07 — 1-в-1. Патч ядра НЕ нужен (он был для legacy `crm.deal.productrows`).
**⚠ Уточнение (проверено на втором портале):** `crm.item.productrow.set` **не пересчитывает**
`opportunity` сущности на порталах **без торгового учёта/каталога** (сумма остаётся 0), и ручное
`opportunity` держится только вместе с `isManualOpportunity:'Y'`. Поэтому crm-sync проставляет
`opportunity`=Σ(`round2(price×(1+rate/100))·qty`, по-юнитно как Bitrix) + `isManualOpportunity:'Y'`
**только для сделки(2)/КП(7)/смарт-счёта(31)** — у смарт-процессов (≥1000) поля может не быть
(создание упало бы), их пропускаем. Также подтверждено: `productrow.set` считает НДС даже при
**пустом** `crm.vat.list` — жёсткий гейт «ставки нет в портале → ошибка» это осознанная политика
учёта, а не техническое ограничение REST. **Решение владельца (2026-07-09): гейт остаётся строгим —
если ставки нет в портале, это ОШИБКА (в чат ошибок), сущность не создаём. «У нас 1-в-1».**

**✅ Сквозная проверка на живом портале (2026-07-09, засеянный тест-портал).** Полный путь пройден
вживую: USD-инвойс → **агент DeepSeek** (`deepseek-v4-flash`, извлёк поставщика+INN+позиции+НДС) →
гейты валюты/НДС → `findCompanyByTaxId` (по `RQ_INN` → компания) → `crm.item.add` (companyId +
opportunity 786 + `isManualOpportunity`) → `crm.item.productrow.set` (позиции персистятся при наличии
каталога) → файл на **общий Диск** → **дело** с файлом (`configurable.add` — только OAuth-контекст,
`ERROR_WRONG_CONTEXT` по вебхуку; фолбэк `todo.add`+`fileTokens=disk<id>`) → **чат** (`im.message.add`).
Артикул-поиск товара по свойству каталога — тоже ✅ вживую (см. «Товар — `crm.product.list`…» выше).

**Eval-скоринг-харнесс не
используем** (решение D5) — фидельность проверяем ручными тест-накладными и юнит-тестами чистого
ядра сборки payload (детерминированная сверка вход→ожидаемый productrow).
