# Мультиязычность документов (редизайн procure-ai)

> Last reviewed: 2026-07-07

**Фиксируем как требование, чтобы не потерялось.** Документы, которые загружают в приложение, бывают
на трёх языках: **русский, белорусский, казахский**. Приложение обязано распознавать и корректно
обрабатывать любой из них — извлечение текста, распознавание налогового ID, классификация типа
документа, единицы измерения. Отдельный перевод не требуется: LLM (DeepSeek) работает с текстом на
любом языке; наша задача — заложить язык-осведомлённость там, где логика зависит от слов.

Связано: поиск поставщика по `RQ_INN` и промпт агента — [`02-target-architecture.md`](02-target-architecture.md)
§«Поиск поставщика» и §«Промпт агента».

---

## 1. Где язык влияет (и где нет)

| Слой | Зависит от языка? | Почему |
|---|---|---|
| Извлечение текста (OCR) | **Да** | tesseract нужны языковые пакеты `rus`+`bel`+`kaz` (+`eng` для латиницы/кодов) |
| Распознавание налогового ID | **Да** | метка идентификатора пишется на языке документа (ИНН/УНП/БИН/БСН…) |
| Классификация типа документа (роутинг) | **Да** | «накладная»/«таварная накладная»/«жүкқұжат» — одно и то же понятие |
| Ключевые слова в правилах роутинга | **Да** | админ задаёт слово; в разных языках слово разное |
| Единицы измерения | **Да** | «шт»/«дана»/«кг» — словарь сопоставления (Q11) |
| Поиск по `RQ_INN` (REST) | **Нет** | поле одно для всех стран; сверяем число |
| Поиск товара по артикулу | **Обычно нет** | артикул — код (язык-нейтрален); по названию — да |

## 2. Налоговый идентификатор — метки и форматы по странам/языкам

Поле в Б24 всегда `RQ_INN`; распознать метку и число в документе — задача агента.

| Страна | Рус. метка | Нац. метка (язык) | Формат |
|---|---|---|---|
| Россия | **ИНН** | ИНН | 10 цифр (юр.) / 12 (физ./ИП) |
| Беларусь | **УНП** | УНП (бел. «УНП» — улікова-нумарацыйны) | 9 цифр |
| Казахстан | **БИН** (юр.) / **ИИН** (физ.) | **БСН** (каз. бизнес-сәйкестендіру нөмірі) / **ЖСН** (жеке...) | 12 цифр |

**Примеры вхождений в документе (что должен поймать агент):**
- рус.: `ИНН 7701234567`, `УНП 190000000`, `БИН 900101000000`
- бел.: `УНП 190000000` (метка та же), `плацельшчык: УНП …`
- каз.: `БСН 900101000000`, `ЖСН 850101000000`, `сатушының БСН-і: …`

**Вывод:** в промпте — таблица меток налогового ID на трёх языках (рус/бел/каз) + форматы (длина/цифры).
Извлечённое число нормализуем (убрать пробелы/разделители) и ищем по `RQ_INN`. Если не распознан/не
найден — путь «поставщик не найден» (сущность создаём без компании + предупреждение).

## 3. Тип документа и ключевые слова (роутинг)

Классификацию типа делает агент (язык-агностично), но **ключевые слова в правилах роутинга** админ
задаёт текстом — и они языкозависимы. Примеры соответствий:

| Понятие | рус. | бел. | каз. |
|---|---|---|---|
| Товарная накладная (ТН) | накладная, товарная накладная, ТН | накладная, таварная накладная | жүкқұжат, тауар жүкқұжаты |
| Счёт (на оплату) | счёт, счёт-фактура | рахунак | шот, шот-фактура |
| Коммерческое предложение (КП) | КП, коммерческое предложение | камерцыйная прапанова | коммерциялық ұсыныс |
| Спецификация | спецификация | спецыфікацыя | ерекшелік |

**Вывод:** (1) в правиле роутинга `keyword` матч — детерминированный, с нормализацией и homoglyph-fold;
(2) один админ обычно работает на одном языке, но приложение должно позволять задать несколько
ключевых слов на правило (список синонимов, в т.ч. на разных языках); (3) классификатор типа в
промпте оперирует понятием, а не буквальным словом, — устойчив к языку.

## 4. Единицы измерения (примеры для словаря, Q11)

| Понятие | рус. | бел. | каз. | код Б24 (`catalog.measure`) |
|---|---|---|---|---|
| штука | шт, штука, шт. | шт, штука | дана, дн | 796 (шт) |
| килограмм | кг | кг | кг | 166 (кг) |
| метр | м | м | м | 006 (м) |
| литр | л | л | л | 112 (л) |

**Вывод:** словарь единиц (Q11) наполняем синонимами на трёх языках → код `catalog.measure`. Нет
соответствия → дефолтная единица / авто-создать + ошибка в чат ошибок.

## 5. Гомоглифы и казахская кириллица (важный нюанс)

- Homoglyph-fold (кир↔лат) уже есть в ядре (`foldHomoglyphs`) — для артикулов/номеров, где путают
  латиницу и кириллицу.
- **Казахская кириллица** содержит буквы, которых нет в русской: **ә, ғ, қ, ң, ө, ұ, ү, һ, і**.
  Нормализация текста и homoglyph-fold **не должны их ломать** (не «схлопывать» `қ`→`к`, `ө`→`о` без
  необходимости) — иначе поиск по названию/ключевым словам исказится. Тесты чистого ядра должны
  включать казахские строки.

---

## 6. Выводы (что закладываем)

1. **OCR:** tesseract с пакетами `rus`+`bel`+`kaz`+`eng` (старый проект был `rus+eng+bel` — добавляем `kaz`).
2. **Промпт агента:** таблицы меток налогового ID и понятий типа документа на рус/бел/каз; извлечение
   и классификация — язык-агностичны; примеры из §2–§3 — в промпт.
3. **Роутинг:** `keyword`-условие принимает список синонимов (в т.ч. многоязычных); матч детерминированный.
4. **Единицы:** словарь с многоязычными синонимами → `catalog.measure`.
5. **Ядро:** нормализация/homoglyph-fold сохраняет казахские буквы; тесты на рус/бел/каз-строках.
6. **1-в-1 со счётом** сохраняется независимо от языка документа.
