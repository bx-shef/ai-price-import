# Политика данных (редизайн procure-ai)

> Last reviewed: 2026-07-22

Что за данные обрабатывает приложение, куда они уходят, где хранятся и на каких условиях. Важно,
потому что приложение работает с **коммерческими данными клиента** (накладные/счета/КП, реквизиты
поставщиков, цены) и облачным приложением Маркета Bitrix24 (мультитенант). Смежное — юрисдикция LLM
в [`03-stack.md`](03-stack.md).

---

## 1. Потоки данных (что куда уходит)

| Данные | Куда | Условие / примечание |
|---|---|---|
| **Текст документа** (после извлечения) | на инференс во **внешний LLM**: по умолчанию **BitrixGPT** (юрисдикцию несёт Битрикс); опц. DeepSeek/КНР по env | только текст; бинарник файла в LLM не уходит. Провайдер — `LLM_PROVIDER` (см. §3, `03-stack.md`) |
| **Исходный файл** | **Диск Б24 клиента** (общий диск → папка приложения → подпапка по месяцам) | остаётся внутри портала клиента; наружу не уходит |
| **OAuth-токены портала** | Postgres backend (издатель) | `refresh_token` шифруется (AES-256-GCM); write-once `application_token` |
| **Счётчики метрик** (пожизненные) | Postgres backend (издатель) | скоуп per-portal; агрегаты для дашборда/алертов (`metrics_counter`) |
| **Трейсы/метрики OTel** (DEFAULT OFF) | внешний OTLP-коллектор (станция телеметрии) | только «форма/счётчики/статус» операций; **никаких** поставщика/артикула/цены/содержимого — тройная защита (см. §4) |
| **Отзывы сотрудников** | Postgres + **закрытый (приватный) репозиторий издателя** (issues, durable outbox) | текст отзыва + контекст; **файл — только с явного согласия** (см. §2) |
| **Сигналы агента** | Postgres/телеметрия (издатель) | уверенность/проблемы; без содержимого документа |

## 2. Согласие на отправку файла (ключевой момент)

При отправке отзыва (👍/👎/💡) сотруднику задаётся явный выбор: **прикладывать исходный файл
документа к отзыву или нет.**

- **По умолчанию — БЕЗ файла.** Уходят только текст отзыва + технический контекст (что распозналось,
  какие нестыковки), без самого документа.
- **Файл уходит в закрытый репозиторий издателя только с явного согласия сотрудника** (чекбокс/кнопка
  «приложить файл»).
- Причина: документ — коммерческие данные клиента; отправка наружу (даже в приватный репо издателя)
  требует осознанного согласия отправляющего.

Формулировку согласия и место (UI отзыва) держим единообразно; текст — на русском, нейтральный.

## 3. Хранение, ретенция, юрисдикция

- **Провайдер по умолчанию — BitrixGPT** (AI Router Битрикс24; уходим от прямого КНР-инференса, #215). ⚠ Владельцу сверить
  точные условия юрисдикции/маршрутизации BitrixGPT с Битрикс (роутер может проксировать к сторонним моделям). Текст документов всё равно уходит на **внешний** LLM-инференс — это указывается клиенту.
  Слой провайдер-агностичен (`LLM_PROVIDER`: bitrixgpt/deepseek/custom) → DeepSeek (КНР) остаётся опцией
  только с согласием клиента/юриста (см. `03-stack.md`).
- **Файлы** живут на Диске клиента; приложение их наружу не выносит (кроме явного согласия §2).
- **Токены** шифруются в покое; удаление приложения (`ONAPPUNINSTALL`) чистит данные портала.
- **Ретенция** метрик/отзывов/логов — задать при реализации (не логировать секреты/полный текст
  документа в общие логи).

## 3a. Self-hosted контур (разворачивание на сервере клиента)

Маркетинг предлагает апселл «развернём на вашем сервере» (см. [04-marketing §2](04-marketing-landing.md)).
Что при этом меняется по данным — **важно не ввести клиента в заблуждение «всё остаётся у нас»**:

- **Остаётся у клиента:** приложение/воркер, Postgres (токены), Redis/очередь, извлечённые данные,
  временные файлы — на его инфраструктуре.
- **ВСЁ РАВНО уходит наружу:** **текст документа на LLM-инференс** внешнему провайдеру (по умолчанию **BitrixGPT** — юрисдикция Битрикса; DeepSeek/КНР — опция). Self-hosted **сам по себе не делает обработку локальной** — периметр покидает
  тот же текст, что и в облаке. Если клиенту нужен полностью замкнутый контур — это **отдельный
  разговор про локальную/он-прем LLM** (не входит в базовый self-hosted апселл; технически слой
  провайдер-агностичен (`LLM_PROVIDER`), но локальную модель нужно поднимать и оплачивать отдельно).
- **Ответственность смещается:** аптайм/бэкапы/патчи/шифрование ключей на инфраструктуре, которую
  издатель не контролирует — зона клиента; условия зафиксировать в оффере (Issue #96, см. §5).
- **РФ/РБ:** для РФ — по референс-модели; для РБ юрисдикцию/хостинг уточнить (§5).

## 4. Телеметрия (OpenTelemetry, DEFAULT OFF)

Глубокая наблюдаемость backend'а — OpenTelemetry (трейсы/метрики по OTLP; детали и env —
[`docs/OBSERVABILITY.md`](../OBSERVABILITY.md)). **По умолчанию выключена** (без
`OTEL_EXPORTER_OTLP_ENDPOINT` бэкенд работает как раньше). Спаны **не несут коммерческих данных**
клиента (поставщик/артикул/цена/содержимое документа) — **тройная защита**:

1. наши ручные спаны эмитят **только allowlist** безопасных ключей (`server/utils/telemetryAttributes.ts`
   `pickSafeAttributes` — поставщика/артикул/цену/содержимое прикрепить нельзя, значение-объект/массив
   отбрасывается);
2. **redaction-SpanProcessor** в бутстрапе (`otel.instrument.mjs`) срезает чувствительные атрибуты
   авто-инструментирования (`db.statement`/`url.*`/`*body*`/`*token*`/…) до экспорта, а `pg` идёт с
   `enhancedDatabaseReporting:false` (значения SQL-параметров не собираются); повторно чистит коллектор
   станции (`transform`-процессор);
3. member_id → необратимый `portal.hash` (SHA-256), `error_kind` — токен из `code`/`name`, **не** текст
   ошибки (`recordException` не вызывается).

⚠ Инвариант: SQL остаётся **параметризованным** — имя pg-спана это неотредактированный канал, и литерал
в непараметризованном запросе попал бы в имя спана. Ретеншн сырья в ClickHouse станции — 72ч.

  условия обновлений/патчей, оффер полностью-локального контура (он-прем LLM). Владелец РФ-модели —
  издатель; РБ — по запросу. Полный чек-лист и ТЗ калькулятора — [11-pricing-selfhosted.md](11-pricing-selfhosted.md).
