План маркетинговой части: публичный лендинг + промо/cross-sell + аналитика. Подход и почти все
компоненты переносимы 1-в-1 из эталона `client-bank-alfa-by` (который сам портирован с
`offer.bx-shef.by`, репо `bx-shef/Lp`). Здесь — что берём, как это ложится на procure-ai и что
меняется под наш домен (закупки/прайсы вместо выписок).

**Решение Q6: публичный лендинг делаем** (как у эталона). **Решение Q4: продукт — облачное
приложение для Маркета Bitrix24**, поэтому лендинг ведёт в двух направлениях: (1) установка из
**Маркета** (основной CTA — карточка `AppInBitrixCard` со ссылкой на листинг) и (2) заявка на
внедрение/доработку (`BriefForm`). Бренд — **нейтральный** (не «Строительный Берег»); издатель
листинга Маркета — **ИП Шевчик И.С.** (Q10). Ниже — план работ.

## 2. Что меняем под procure-ai

- **Контент `landing.ts`** — переписать под домен закупок: боль «менеджер вбивает накладные руками» →
  результат «загрузил файл — сделка в „Закупках“ создалась сама». Шаги «Как это работает» (3):
  загрузка файла → AI распознаёт поставщика, позиции и цены → сделка в Bitrix24. «Почему мы»:
  распознавание PDF/сканов/фото/XLSX/DOCX, поиск по УНП/артикулу, позиции 1-в-1 со счётом,
  безопасность (изолированный MCP, недоверенный документ). Подбор договора — как отдельная
  индивидуальная доработка (не в базовом флоу).
- **Форматы/интеграции** (tech-row) — PDF, скан/фото (OCR), XLSX/XLS, DOCX; таргет — Bitrix24
  (воронка «Закупки»), на горизонте 1С:УТ.
- **Hero-анимация** — вместо node-graph `HeroGraph` сделан `HeroParticles.vue`: сдержанное
  parallax-поле точек с редкими вспышками связей (владелец просил «совсем другую», не хаб-граф).
  Canvas, cyan, reduced-motion/пауза/троттлинг/aria-hidden.
- **OG/бренд** — свой заголовок/сабтайтл и, при необходимости, партнёрский бейдж (ИП/интегратор).
- **Промо `CustomDevCard`** — ссылка на бриф/партнёра (как в экосистеме `bx-shef`), тексты вшиты.
- **`AppInBitrixCard` — центральный CTA** (не второстепенный, как в эталоне): ссылка на **листинг
  Маркета Bitrix24** + мобильный QR листинга. Это основной путь установки облачного приложения.
- **Блок «Индивидуальная доработка под ваш процесс»** — платная фича поверх стандартного приложения
  (через `CustomDevCard`/`BriefForm`): «настроим на вашем сервере подбор **договора**, сопоставление
  **по артикулу**, **загрузку вне портала** (standalone / приём по e-mail / подхват из папки по
  **SFTP** — на сервере клиента) и другие правила под ваш учёт». Сюда уходят убранные из стандарта
  фичи — подбор договора (Q8) и внепортальная загрузка (стандартное приложение работает **только
  внутри портала Б24**, в т.ч. в **мобильном приложении**) — как продающее предложение, а не ядро продукта.
- **Позиционирование по скорости — «стабильно, а не быстро» (важно, вшить в тексты и калькулятор).**
  Облачное приложение работает на **общей минимальной инфраструктуре** с **ограниченным параллелизмом**
  (нагрузка всех арендаторов делит скромный сервер) — это осознанный выбор (мы не вкладываемся в
  производительность общего сервера, см.
  [09-deploy §«Ресурсы воркера»](09-deploy.md#ресурсы-воркера-минимум--стабильность-осознанный-выбор)).
  Честный месседж клиенту: **распознавание надёжное, но не мгновенное**; в пик очередь может подождать.
  Не обещаем «за секунды» — обещаем «сам, точно, без ручного ввода».
  - **Апселл «нужно быстро / большие объёмы → развернём на вашем сервере» (self-hosted).** Выделенные
    ресурсы клиента = параллельная обработка **без соседей-арендаторов в общей очереди** (сама
    очередь BullMQ/Redis остаётся — просто без конкуренции чужих тенантов). Модель разворачивания
    «под клиента» — по образцу внутреннего референса «чёрная дыра» (вайбкодинг-Битрикс24) для **РФ**.[^blackhole]
    ⚠️ Открытые вопросы перед выносом в оффер: кто хостит/патчит/бэкапит инстанс, SLA, как едут
    обновления образа; и — **данные не остаются целиком у клиента**: текст документа всё равно уходит
    на LLM-инференс внешнему провайдеру (см. [05-data-policy §self-hosted](05-data-policy.md)). Для
    **РБ** юрисдикцию/условия/хостинг нужно **уточнить** (TODO) — не копировать РФ-модель вслепую.
    Идёт тем же каналом, что индивидуальная доработка (`CustomDevCard`/`BriefForm`).
  - **Калькулятор расчётов:** добавить тариф/опцию **«на вашем сервере» (self-hosted)** рядом с
    облачной подпиской — с явной развилкой «облако (ограниченный параллелизм, дешевле) ↔ свой сервер
    (быстро, выделенно)». Единицу расчёта (за документ / за пользователя / фикс) — определить отдельно.
    Для РФ — по референс-модели; для РБ — плейсхолдер «по запросу», пока условия не уточнены.
    (Калькулятора как компонента пока нет — это требование к нему, а не готовая цена; в бэклоге: docs/ROADMAP §4.0.)
    Полное ТЗ калькулятора + чек-лист self-hosted оффера — [11-pricing-selfhosted.md](11-pricing-selfhosted.md).

[^blackhole]: «Чёрная дыра» — **внутренняя референс-модель** self-hosted-разворачивания (по опыту
    вайбкодинг-Битрикс24), а НЕ клиентская формулировка. В публичные тексты/оффер/калькулятор не
    переносить дословно — описывать нейтрально («разворачивание на выделенном сервере клиента»).

## 3. Технические инварианты (из эталона, не ломать)

- **Форс-dark только на `/`** через `htmlAttrs data-force-dark` + `.landing-shell`; in-portal
  страницы (`/app`,`/import`,`/install`,`/metrics`) остаются light/dark-auto. FOUC-гард — `theme-init`
  в `app.vue` до первого рендера.
- **CSP** — заголовком в `nginx.conf`. `script-src` держит `'unsafe-inline'`: страницы отдаёт
  backend Nitro (nginx проксирует `/`), а Nuxt пере-инъектит `runtimeConfig.public` в инлайн
  `window.__NUXT__.config` **в рантайме** из `NUXT_PUBLIC_*` контейнера — поэтому sha256-хэш,
  посчитанный на сборке, на отданном скрипте не совпадёт, браузер заблокирует бут Nuxt, гидратация
  не пройдёт и client-only компоненты (демо) не смонтируются. Инлайн-хэши несовместимы с рантайм-
  инъекцией конфига здесь. `frame-ancestors`/`connect-src` разрешают облачные домены Б24.
- **Форма Б24 — только через `public/b24-form.html`** с form-scoped CSP (`location = /b24-form.html`),
  чтобы официальный загрузчик работал, а строгий CSP страницы не ослаблялся. URL строит чистый
  `b24Form.ts` (allowlist хостов + валидация id/secret; тесты). Пустой конфиг ⇒ слот-плейсхолдер.
- **Метрика в iframe отключается** (`window.self !== window.top`) — иначе webvisor писал бы
  session-replay CRM клиента, а цели пачкали бы аналитику лендинга портальным трафиком.
- **Контент централизован** в `landing.ts` и оттуда же кормит SEO — единственный источник правды.

## 4. Definition of Done (визуальная верификация)

После любой правки UI/лендинга — `pnpm generate && pnpm screenshot`, смотреть `screenshots/`
(mobile/desktop × light/dark) до того, как считать готовым. OG перегенерировать при смене
заголовка/брендинга (`pnpm og`).

## 5. Порядок работ (этап 3 карты проекта)

1. Порт `landing.vue` + `main.css` (`.landing-shell` токены) + шрифты.
2. Порт `landing.ts` → переписать контент под закупки; подключить SEO в `app.vue`.
3. Hero-анимация `HeroParticles.vue` (parallax-поле; вместо node-graph эталона).
4. Порт `BriefForm.vue` + `b24Form.ts` + `public/b24-form.html` + nginx form-scoped CSP.
5. Порт промо (`CustomDevCard`, `AppInBitrixCard`) + визитка (если нужна).
6. Метрика (`nuxt.config.ts` инлайн + `useMetrikaGoal`); CSP — заголовком в `nginx.conf`.
7. OG-генерация (`make-og.mjs`) под свой бренд.
8. Скриншоты, пиксель-ревью, обновить `Last reviewed`.
