# План тестирования procure-ai

> Last reviewed: 2026-07-09

Трёхуровневый план проверки редизайна: от «вообще стартует» до «протестировано со всех сторон».
Уровни идут по возрастанию охвата и стоимости — проходим их по порядку, каждый следующий имеет
смысл только когда предыдущий зелёный.

- **Уровень 1 — «просто запустить»** (smoke): за пару минут убедиться, что ничего не падает на
  старте. Внешних сервисов (Redis/Postgres) не нужно.
- **Уровень 2 — «просто прогнать»** (happy-path): один раз пройти штатные сценарии. Часть требует
  `docker-compose` (Redis+Postgres) и/или живого тестового портала Б24.
- **Уровень 3 — «со всех сторон»** (полное покрытие): граничные случаи, безопасность,
  отказоустойчивость, регрессия.

Команды и роуты — из `package.json` и `server/api/*`; живой REST проверяем по вебхуку `B24_HOOK`
(скоупы `crm, catalog, disk, im, placement`), а не по памяти.

## Уровень 1 — просто запустить (smoke)

Цель — стартует ли вообще. Критерий прохождения: всё поднялось, health отвечает, страницы не белые.

| Шаг | Команда / действие | Что ждём |
|---|---|---|
| Статика собирается | `pnpm generate` | Сборка без ошибок, страницы предрендерятся |
| Дев-сервер поднимается | `pnpm dev` | Стартует на `:3000`, в консоли нет ошибок |
| Лендинг открывается | `/` | Тёмный лендинг, анимация hero, кнопки |
| In-portal страницы рендерятся вне фрейма | `/app`, `/import`, `/settings`, `/login`, `/queues` | Отдаются без белого экрана (SDK молча no-op вне портала) |
| Backend жив | `GET /api/health` | `{ status, time, commit }` |

## Уровень 2 — просто прогнать (happy-path)

Цель — прогнать основной сценарий один раз. Критерий: один документ проходит весь путь до записи
в CRM, уведомление в чат приходит.

| Область | Действие | Что ждём |
|---|---|---|
| Авто-проверки | `pnpm check` (lint + typecheck + test) | Всё зелёное (сейчас 333 теста) |
| Ручная загрузка | на `/import` перетащить прайс/накладную | Файл разобрался, предпросмотр позиций |
| Очереди | `docker-compose up`, `/queues?preview=1` | График очередей рисуется |
| Пайплайн | документ → `file-extract → agent-run → crm-sync` | Джоба доходит до конца, статус `done` |
| Живой REST (вебхук) | точечно: диск (папка+файл), `im.message.add`, `crm.product.list` | Отрабатывают (проверено вживую) |
| Установка в портал | `/install` на тестовом портале | `event.bind` + `installFinish`, токен на backend |

## Уровень 3 — со всех сторон (полное покрытие)

Цель — граничные случаи, безопасность, отказоустойчивость, регрессия. Критерий: все граничные
случаи ведут себя предсказуемо (ошибка → в чат ошибок, без потери строк), безопасность закрыта,
после падений система восстанавливается.

### Функционально / данные

- Форматы входа: PDF-текст, скан (OCR рус/бел/каз), office, `1CClientBankExchange`, client-bank `^Type=`.
- НДС: с НДС / без НДС / разные ставки по строкам / «включён ли НДС» не определён → жёсткая ошибка
  (не создаём кривую сущность, не теряем строки).
- Товар: по артикулу (попал / не попал + фолбэк на имя), `onMissing` = skip-warn / create / freeform.
- Поставщик: найден / не найден (создаём без компании + предупреждение).
- Валюта не в портале → ошибка в чат, сущность не создаём.
- Идемпотентность: повтор джобы → не дублирует сущность и не шлёт второй раз «успех».

### Безопасность

- BB/промпт-инъекция в назначении/имени → нейтрализация перед чатом (`[url]`, упоминания, кнопки).
- Мультитенант-изоляция: чужой `member_id` / чужой `app.option` недостижимы.
- Агент (DeepSeek) не видит токенов Б24; sanitized env; запрещённые тулы.
- Загрузка: превышение размера → 413; нет Redis → 503; traversal в пути файла.
- Операторская авторизация: неверный пароль, брутфорс-троттлинг (`limit_req`), подпись cookie.

### Отказоустойчивость / инфраструктура

- Redis/Postgres упал и поднялся; ретраи джоб; синхронный фолбэк приёма событий Б24.
- Удаление приложения → чистка данных портала (токены, дедуп, файлы).
- TTL-выметание старых загрузок.
- Docker: сборка `backend`-образа с бинарями извлечения; `healthcheck`; `docker-compose.prod`.

### Регрессия / визуал

- `pnpm check` в CI; сборка Docker-образа на каждом PR.
- Визуальная проверка страниц (свет/тёмная, мобайл/десктоп).
- Прогон реальных фикстур выписок/прайсов.

## Что уже покрыто автоматически

Юнит-тесты (Vitest, 2 проекта — `unit` node + `nuxt`) держат чистое ядро: НДС, единицы, роутинг,
дедуп, гомоглифы, разбор форматов, поиск товара/компании, чат-уведомления (BB-защита),
идемпотентность `crm-sync`, приём событий Б24. Прогон — `pnpm test`; полный гейт — `pnpm check`.
