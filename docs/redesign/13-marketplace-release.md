      согласием). ⚠ В карточке/политике всё равно честно указать: текст документа уходит на внешний
      LLM-инференс (BitrixGPT). Провайдер задаётся `LLM_PROVIDER` (дефолт в коде — bitrixgpt).
- [ ] **Опубликована политика конфиденциальности** по постоянному HTTPS-URL (см. §2). **Черновик готов —

      юристом, опубликовать по URL. Основа — [`05-data-policy.md`](05-data-policy.md).
- [ ] **Прод стабилен** — `price-import.bx-shef.by` поднят, `GET /api/health` = ok, установка/удаление

## 3. Тариф (решение владельца — сейчас черновик)

`11-pricing-selfhosted.md` — **черновик требований, цены не утверждены**. Перед листингом зафиксировать:

- [ ] **Единица тарификации**: за портал (подписка) / за пользователя / за N документов / фикс. **Не решено.**
- [ ] **Бесплатный триал** (напр. 14 дней или N импортов) — снижает барьер, помогает модерации/отзывам.
- [ ] **Облако vs self-hosted**: облачная подписка в Маркете + **платный self-hosted** для скорости/объёма
      и доработок (подбор договора Q8, артикул-матчинг, не-портальный ingestion). Self-hosted **всё равно**
      шлёт текст во внешний LLM — честно указать.
- [ ] Калькулятор цены на лендинге — **компонента пока нет** (бэклог ROADMAP §4.0).

## 4. Процесс подачи на модерацию

1. Кабинет разработчика → создать приложение (тип: **тиражное/облачное**, не локальное).
2. Заполнить карточку (§2), указать OAuth `client_id`/`client_secret` → прописать в прод-env
   (`B24_CLIENT_ID/SECRET`) и **передеплой** (env читается рантаймом; `NUXT_PUBLIC_SITE_URL` — на build).
3. Прогнать установку на **своём** портале из черновика листинга (не по прямой ссылке) — проверить
   полный цикл: установка → настройка маппинга → импорт → удаление (чистка данных).
4. Отправить на **модерацию** Bitrix24. Готовить ответы на типовые вопросы модерации: куда уходят
   данные (внешний LLM!), какие скоупы и зачем, как чистятся данные при удалении.
5. После одобрения — публикация. **Не публиковать раньше гейтов §0.**
