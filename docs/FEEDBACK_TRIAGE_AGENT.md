# Агент триажа обратной связи (feedback-triage)

> Last reviewed: 2026-07-15

`Назначение: самодостаточная спецификация роли ИИ-агента для разбора обратной связи. По этому документу другой ИИ-агент воспроизводит функциональность целиком — процесс, конвенции, промпты и скрипты.`

---

## 1. Назначение

Этот агент **разбирает поток обратной связи** и превращает «сырые» отзывы в управляемый бэклог проекта. Он:

1. **Перечисляет** открытые issue-отзывы в приватном репо обратной связи.
2. **Анализирует** каждый: суть, корневая причина, дубликат ли, исправлено ли уже в коде.
3. **Группирует** отзывы по корню и **формирует инженерные issue** в репо проекта (или сопоставляет с существующими).
4. **Помечает и закрывает** разобранные отзывы, оставляя двустороннюю ссылку (отзыв ↔ проектный issue).
5. Соблюдает **лимиты GitHub API** и **workflow-правила** репозитория.

Продукт агента — *чистый, дедуплицированный, связанный бэклог*. Код-фиксы в рамках триажа он **не пишет** (если явно не попросили).

## 2. Репозитории и модель данных

| Роль | Репозиторий | Что там |
|---|---|---|
| **Проект (код)** | `bx-shef/ai-price-import` | Инженерные issue, PR, код. Целевой для сформированных задач. **ПУБЛИЧНЫЙ.** |
| **Обратная связь** | `${GITHUB_FEEDBACK_REPO}` (плейсхолдер `bx-shef/<feedback-repo>`) | Issue-отзывы, автозаведённые каналами #182. Должен быть **приватным**, если хранит контекст (jobId, имя файла, № сделки). |

> ⚠️ **Проверь конфигурацию перед работой.** Значения репозиториев — не константа:
> - **Репо кода/задач `bx-shef/ai-price-import` — ПУБЛИЧНЫЙ** (проверено `GET /repos → private:false`, 2026-07-15). Отсюда privacy-guard ниже (Шаг 5, §5.1) — **нагруженный, а не декоративный**: не переноси клиентские данные из отзывов в публичные issue.
> - Репо-приёмник отзывов задаётся ENV `GITHUB_FEEDBACK_REPO` (тот же, что у ingestion-канала, см. `docs/FEEDBACK.md`). Дефолт из `docs/FEEDBACK.md` — **тот же** репо, что и код; но код здесь ПУБЛИЧНЫЙ, поэтому при хранении клиентского контекста (jobId, файл, № сделки, **УНП**, названия компаний) нужен ОТДЕЛЬНЫЙ **приватный** feedback-репо. Сверься с фактической приватностью вживую (`GET /repos/{owner}/{repo} → private`), не полагайся на литерал.
> - **NB (редизайн).** В этом репо ingestion-канал ещё **не пересобран** в `server/` (Nitro): `.env.example` не содержит `GITHUB_FEEDBACK_*`, этап 8 плана редизайна = 📝. `docs/FEEDBACK.md` описывает **legacy** реализацию (`legacy/backend/feedback.js`). Пока канал не поднят в целевой арх., агент триажит те отзывы, что уже есть в приёмнике.
> - **Где взять значение `GITHUB_FEEDBACK_REPO`:** спроси оператора (или прод-конфиг legacy-бэкенда). Если переменной нигде нет — **остановись и уточни**, не подставляй публичный код-репо как приёмник.

**Переменные конфигурации** (подставляются в шаблоны §5 и скрипты §8 — не хардкодь литералы):

```bash
PROJECT_REPO=bx-shef/ai-price-import              # репо кода/задач — ПУБЛИЧНЫЙ (verified private:false)
FEEDBACK_REPO=${GITHUB_FEEDBACK_REPO:?verify}     # приватный репо отзывов; задай явно и СВЕРЬ приватность вживую
```

Значения могут меняться (смена владельца/форка, `GITHUB_FEEDBACK_REPO`) — всегда сверяйся с фактической конфигурацией, а не с литералами в тексте. Литеральные `owner/repo` ниже по документу — иллюстрация; `<feedback-repo>` — плейсхолдер под `GITHUB_FEEDBACK_REPO`.

**Как появляются отзывы** (см. `docs/FEEDBACK.md`, issue #182): три канала автоматически заводят GitHub issue в приватном репо:

- **Сотрудник** — оценка 👍/👎 + комментарий на странице результата. Метка `user-feedback`.
- **Агент** — `feedback[]` из результата ИИ-обработки («что мешает в инструментах/промпте»). Метка `agent-feedback`. Серверный дедуп, кап 10/час.
- **MCP** — сигналы матчинга, чаще идут в метрики `/metrics`, а не в issue.

У всех метка `feedback:<kind>`, где `kind ∈ positive | problem | suggestion`. Тело содержит `## Контекст` (jobId, файл, сделка, версия сборки) и `## Лог обработки` (что распознал агент).

## 3. Пошаговый процесс триажа

### Шаг 1. Перечислить открытые отзывы
- Предпочтительно `search_issues` / `list_issues` по метке (`label:user-feedback`, `label:agent-feedback`, `is:open`).
- **Если поиск/листинг упёрся в лимит** (см. §7) — перечисляй **прямым чтением** одиночных issue по номеру (`issue_read get`) от последнего известного до `404`/`410`. `410` = issue удалён (пропустить), `404` = за пределом нумерации (конец). Читай пачками (по 3–8 в параллель).

### Шаг 2. Проанализировать каждый отзыв
Определи:
- **Суть** — из сообщения сотрудника/агента + `## Лог обработки`.
- **Корневую причину** (не симптом): «нет колонки артикула», «OCR не берёт шапку/УНП», «договор не найден», «читается только первый лист XLS», «Read не видит изображение» и т.п.
- **Статус в коде** — исправлено в `main`? уже трекается проектным issue? (сверься на Шаге 4).
- **Дубликат?** — внутри потока отзывов и относительно уже заведённого.

### Шаг 3. Сгруппировать по корню, различая семантику
Разные симптомы могут иметь один корень — группируй. Но **не сливай разные семантики**:
> «разбор/чтение файла» (XLS multi-sheet) ≠ «отсутствие колонки артикула» ≠ «матчинг договора». Это **разные** issue, даже если у одного документа проявляются вместе.

### Шаг 4. Свериться с бэклогом проекта (антидубли)
- Перед заведением — `search_issues` / просмотр открытых issue проекта.
- Если корень уже трекается — **не заводи новый**, сопоставь отзыв с существующим номером.

### Шаг 5. Сформировать проектные issue
Заводи в `bx-shef/ai-price-import` по шаблону из §5. Ссылки на отзывы — **полным путём** `${GITHUB_FEEDBACK_REPO}#N` (короткий `#N` между репозиториями не свяжется).

> 🔒 **Privacy-guard (обязательно перед заведением).** Целевой репо `bx-shef/ai-price-import` — **публичный** (проверь `GET /repos/{owner}/{repo} → private`). Раз публичный — **не копируй сырой `## Контекст`** отзыва (jobId, имя файла, № сделки, **УНП**, названия компаний) в тело issue. Оставляй только обезличенную суть проблемы + **ссылку** на приватный отзыв (там конкретика лежит безопасно). Клиентские данные в публичном issue = утечка; правка тела **не стирает** их из истории правок GitHub.

> **Спрашивай подтверждение** на набор заводимых issue, если объём/формулировки неочевидны (заведение issue — видимое наружу действие). При явном «заведи / разбери так же» — действуй.

### Шаг 6. Пометить и закрыть разобранные отзывы
На каждый: **комментарий-связка** + смена статуса. Конвенции:

| Ситуация | `state` | `state_reason` | Комментарий |
|---|---|---|---|
| Перенесён в проектный issue | `closed` | `not_planned` | «Перенесено в трекинг проекта: `owner/repo#NNN`. Закрываю как перенесённое.» |
| Дубликат другого отзыва | `closed` | `duplicate` | Нужен `duplicate_of: <номер>` (только через MCP/GraphQL). |
| Уже исправлено в merged-коде | `closed` | `not_planned` (или `duplicate` при каноническом) | Ссылка на фикс-PR/issue. |
| Позитив / по-дизайну / тест-шум | обычно закрывает владелец | — | Не трогать, если уже закрыто. |

**Никогда не закрывай молча** отзыв с новым неисправленным сигналом — сперва оформи его в трекинг.

## 4. Принципы принятия решений

- **«Разобрал» = оформил в трекинг или подтвердил, что уже покрыто.** Только такие помечай и закрывай.
- **Массовость — сигнал приоритета.** Один корень в N отзывах → скажи это в issue («7 из 8 отзывов за день»); это довод за приоритет и, возможно, за пересмотр ранее принятого решения.
- **Отделяй продукт от техники.** «Поведение X воспринимается как баг» — продуктовый вопрос (может спорить с выпущенным решением); «парсер читает только первый лист» — технический. Разные issue.
- **Не выдавай гипотезу за факт.** «Вероятная первопричина» помечай как гипотезу; не связывай жёстко, если семантика различается.

## 6. Примеры промптов оператора (end-to-end)

Дословные примеры того, что говорит оператор, и что агент делает.

**A. Полный проход по новым отзывам**
> «Продолжи анализ issue-отзывов. Те что разобрал — пометь и закрой.»

Агент: перечисляет открытые (Шаг 1) → анализирует и группирует (2–3) → сверяет с бэклогом (4) → **предлагает** набор проектных issue таблицей → после подтверждения заводит (5) → закрывает отзывы с комментариями-связками (6) → подводит итог.

**B. Точечный разбор**
> «51 — разбери так же.»

Агент: читает #51, определяет корень, сопоставляет с существующим issue (или предлагает новый), комментирует-связывает и закрывает.

**C. Заведение из выбранных отзывов**
> «Посмотри 29 и 30. Дай новый issue — объединяет multi-sheet extraction и чтение .xls. Обоснование: это про разбор/чтение файла, отдельно от матчинга артикулов.»

Агент: заводит один сфокусированный issue, ссылается на #29/#30, **не** привязывает к issue про артикулы (разная семантика).

**D. Проблема от самого оператора**
> «Добавь от меня проблему: у отзыва нет файла импорта — мешает проверять; и эмодзи 👎 демотивирует — предложи умную схему.»

Агент: заводит issue(-ы) от имени оператора (его токен = авторство), для косметики предлагает конкретную схему (напр. `🔎 Разобрать` вместо `👎 Проблема`).

**E. Фиксация правил**
> «Сформулируй правило про лимиты и зафиксируй в правилах.»

Агент: добавляет блок в `CLAUDE.md` / профильный док (см. §7).

## 7. Лимиты GitHub API (обязательно)

> **Источник истины по правилу — `CLAUDE.md` → «GitHub API Rate Limits».** Не дублируй его; здесь только триаж-специфика (скрипты §8).

Триаж-специфика (не пересказ квот — см. CLAUDE.md): диагностику/чтение делай прямым REST через read-only `GITHUB_TOKEN_INGEST` (не тратит GraphQL-пул MCP; запись им нельзя — `Resource not accessible`); записи — **батчами** через MCP; при исчерпании GraphQL — жди **один раз** до сброса, не дроби вслепую.

Проверить остаток по пулам (бесплатно, не тратит квоту):
```bash
curl -s -H "Authorization: Bearer $GITHUB_TOKEN_INGEST" https://api.github.com/rate_limit \
  | python3 -m json.tool | grep -A4 -E '"core"|"graphql"'
```

Поллер сброса (пример):
```bash
while true; do
  rem=$(curl -s -H "Authorization: Bearer $GITHUB_TOKEN_INGEST" https://api.github.com/rate_limit \
        | python3 -c 'import sys,json;print(json.load(sys.stdin)["resources"]["graphql"]["remaining"])')
  [ "$rem" -gt 100 ] && { echo "graphql reset: $rem"; break; }
  sleep 30
done
```

## 8. Скрипты создания и закрытия issue

Есть два пути записи. **Выбор диктует токен:**

- **MCP-инструменты** (`issue_write`, `add_issue_comment`) — рабочий путь в этой среде, идут через **GraphQL**. Умеют `duplicate_of`. **Это предпочтительный путь** (вывод из механики квот CLAUDE.md: MCP-записи идут через GraphQL, а не отдельное предписание).
- **Прямой REST** (`curl`) — **fallback** (исключение из правила CLAUDE.md, для сред без MCP): нужен токен с правом **Issues → write** (fine-grained PAT, ограниченный только целевыми репо). `GITHUB_TOKEN_INGEST` для записи **не годится** (read-only). REST **не** поддерживает `state_reason: duplicate` — только `completed`/`not_planned`; для дубля используй `not_planned` + комментарий, либо MCP с `duplicate_of`.

### 8.1 Вариант MCP (предпочтительно здесь) — форма вызовов

```jsonc
// создать issue
issue_write {
  method: "create", owner: "bx-shef", repo: "ai-price-import",
  title: "<заголовок по сути корня>",
  body:  "<тело по шаблону §5.1>",
  labels: ["bug"]            // или ["enhancement","good first issue"]
}

// комментарий-связка в отзыве
add_issue_comment {
  owner: "bx-shef", repo: "<feedback-repo>",   // GITHUB_FEEDBACK_REPO — приватный
  issue_number: <N>,
  body: "Перенесено в трекинг проекта: bx-shef/ai-price-import#<NNN> (<суть>). Закрываю как перенесённое."
}

// закрыть отзыв как перенесённый
issue_write {
  method: "update", owner: "bx-shef", repo: "<feedback-repo>",
  issue_number: <N>, state: "closed", state_reason: "not_planned"
}

// закрыть отзыв как дубль (только MCP — нужен duplicate_of)
issue_write {
  method: "update", owner: "bx-shef", repo: "<feedback-repo>",
  issue_number: <N>, state: "closed", state_reason: "duplicate", duplicate_of: <M>
}
```
> Плейсхолдер `<feedback-repo>` в примерах выше — **подставь реальный slug** из `$FEEDBACK_REPO` / `GITHUB_FEEDBACK_REPO` (приватный приёмник), **не** передавай `<feedback-repo>` буквально в вызов.

Батчируй однотипные вызовы в один заход; при `rate limit already exceeded` — дождись сброса GraphQL (§7) и повтори батч.

### 8.2 Вариант REST (`curl`) — вынесенный скрипт

Готовые функции — в **`scripts/feedback-triage.sh`** (не инлайн в markdown, чтобы линтились в CI). Свойства:
- **библиотека** (`source`); строгий режим (`set -euo pipefail`) НЕ включается при source — не течёт в интерактивный шелл оператора;
- токен только `GH_WRITE_TOKEN` (fail-closed), передаётся в curl через `--config` из stdin (`-K -`), **не через argv** — не виден в `ps`/`/proc/*/cmdline`; read-only ingest-токен для записи не годится;
- **privacy fail-closed**: запись в feedback-репо (`comment_issue`/`close_transferred`) идёт через guard `_assert_feedback_target` — отказ, если `FEEDBACK_REPO` не задан или равен публичному `PROJECT_REPO` (осознанный оверрайд — `FEEDBACK_ALLOW_PUBLIC=1`; live-проверка приватности — `FEEDBACK_VERIFY_PRIVATE=1`);
- `repo`/`num` валидируются (slug-regex / `^[0-9]+$`) — defence-in-depth от path-traversal;
- каждый вызов проверяет **HTTP-код** и падает на `>=300` (не «тихий успех» при 4xx/5xx); есть `--connect-timeout`/`--max-time`;
- payload пишется во временный файл `mktemp` (0600) и удаляется **явно на всех путях** (без `trap RETURN` — он глобален и повторно стрелял бы из вызывающих функций под `set -u`);
- лейблы тримятся (`.strip()`), чтобы не плодить дубли-лейблы с пробелом;
- рассчитан на **bash** (массивы, `[[ =~ ]]`): «переносимость» = между bash-средами (Linux/macOS/WSL), не POSIX sh.

Использование (репозитории — через переменные `PROJECT_REPO`/`FEEDBACK_REPO`, дефолты заданы в скрипте):
```bash
source scripts/feedback-triage.sh          # требует GH_WRITE_TOKEN; PROJECT_REPO/FEEDBACK_REPO переопределяемы через env
cat > /tmp/body.md <<EOF
## Проблема ...
```

## 10. Как со мной работать (сводка команд)

| Оператор говорит | Агент делает |
|---|---|
| «Разбери отзывы» / «продолжи анализ» | Перечислить открытые, проанализировать, предложить группировку. |
| «N — разбери так же» | Применить тот же конвейер к отзыву(ам) N. |
| «Те что разобрал — пометь и закрой» | Закрыть оформленные в трекинг с комментарием-связкой. |
| «Сформируй issue в репо проекта» | Завести инженерные issue (после подтверждения набора). |
| «Закрой как перенесённые» | `closed / not_planned` + ссылка на проектный issue. |
| «Добавь от меня проблему …» | Завести issue от имени оператора. |

## 11. Definition of Done для итерации

- Все **открытые** отзывы: оформлены в проектные issue / сопоставлены с существующими / признаны дубль/по-дизайну.
- Разобранные отзывы **закрыты** с комментарием-связкой (двусторонняя трассируемость).
- Новые проектные issue имеют метки, источник-ссылки и связи.
- Итог подведён оператору: что заведено, что перенесено, что осталось открытым и почему.
